20:T3ea4,<p>We have resources, but need to know how to assign these to the jobs that need them. This note presents some of the most common resource management policies for cloud clusters.</p>
<h3 id="introduction-to-cluster-management">Introduction to cluster management<a class="anchor" aria-hidden="true" hidden href="#introduction-to-cluster-management">#</a></h3>
<p>How can we <strong>allocate</strong> the resources in a cluster in an efficient manner? How can we allocate resources fairly?</p>
<h4 id="two-step-allocations">Two step allocations<a class="anchor" aria-hidden="true" hidden href="#two-step-allocations">#</a></h4>
<p>There are two main kinds of allocation: first you need to allocate resources to a process, then allocate the process physically in the cluster. We have seen an example of a working infrastructure in <a href="/notes/cluster-resource-management/">Cluster Resource Management</a>.</p>
<h4 id="private-and-public-cluster-management">Private and public cluster management<a class="anchor" aria-hidden="true" hidden href="#private-and-public-cluster-management">#</a></h4>
<p>Cluster management could be private or public.</p>
<ul>
<li><strong>Private</strong>: means every app is managing their own <strong>sub-cluster</strong>: each app receives a private, <strong>static</strong> set of resources. Here it is easier to manage hardware for various needs.
<ul>
<li>We have easy <strong>performance isolation</strong></li>
<li><strong>Simple to manage:</strong> each application decides how to use their dedicated hardware.</li>
<li>Often not optimal -> Low resource utilization.</li>
<li>Today this is mostly used for user-facing scale-out workloads (like web servers, etc.)</li>
</ul>
</li>
<li><strong>Public</strong> means there is a big cluster, like standard third party, most of the services we mentioned in <a href="/notes/cloud-computing-services/">Cloud Computing Services</a> are ran in public clusters.
- <strong>Performance isolation</strong> is difficult to achieve.
- <strong>Resource utilization</strong> is higher: resources are shared among multiple applications.
- <strong>Resource management</strong> is more complex: need to manage the resources of the entire cluster.</li>
</ul>
<h4 id="desiderata-for-cluster-management">Desiderata for Cluster Management<a class="anchor" aria-hidden="true" hidden href="#desiderata-for-cluster-management">#</a></h4>
<ul>
<li><strong>Fairness</strong>
<ul>
<li>Users should be granted resources proportional to how much they are paying • Flexibility to express priorities</li>
</ul>
</li>
<li><strong>Efficient resource usage</strong>
<ul>
<li><strong>Work conservation</strong>: Resources should not idle while there are users whose demand is not fully satisfied</li>
</ul>
</li>
<li><strong>Performance isolation</strong>
<ul>
<li>Guarantee that misbehaved collocated applications cannot affect me “too much”</li>
<li>Experience says this is very hard to achieve, especially in public clusters, and with virtual CPUs. You need to have the whole machine to be able to get more performance isolation.</li>
</ul>
</li>
</ul>
<p>Classical resources are:</p>
<ul>
<li>CPU allocation</li>
<li>Memory allocation</li>
<li>Network allocation</li>
<li>Memory bandwidth allocation and similars.</li>
</ul>
<h4 id="theoretical-properties-for-schedulers">Theoretical properties for Schedulers<a class="anchor" aria-hidden="true" hidden href="#theoretical-properties-for-schedulers">#</a></h4>
<p>This section shortly lists some common properties for schedulers in theoretical analysis.</p>
<p><strong>Envy-Freeness Schedulers</strong>
An envy-freeness scheduler is a scheduler that does not allow users to envy each other. This means that no user prefers another user’s allocation over their own. DRF is an example of Envy-free scheduler.</p>
<p><strong>Strategy-Proof Schedulers</strong>
A strategy-proof scheduler is a scheduler that does not allow users to gain an advantage by lying about their demands. This property allows to avoid users trying to game the system by requesting more resources than they actually need. For example, DRF explained later is not strategy-proof. While Max-Min scheduling is.</p>
<p><strong>Work Conserving Schedulers</strong>
A work conserving scheduler is a scheduler that does not leave idle resources when there are users with unsatisfied demands. It is a very important property, because it allows to maximize the resource utilization.</p>
<h3 id="max-min-scheduling">Max-Min Scheduling<a class="anchor" aria-hidden="true" hidden href="#max-min-scheduling">#</a></h3>
<h4 id="the-method">The method<a class="anchor" aria-hidden="true" hidden href="#the-method">#</a></h4>
<p>The algorithm is quite easy:</p>
<ul>
<li>Allocate resources in order of increasing demand</li>
<li>No user gets a resource share larger than its demand</li>
<li>Users with unsatisfied demands get an equal share of the resource (remember you need to <strong>weight</strong> this! so if one user has double weight, it gets double of remaining shares)</li>
</ul>
<p>The problem is when you have different kinds of resources to share. Another problem is that it is not always predictable to know how many tokens are you getting, this is mostly dependent by what others require.
It is easy to extend this allocation method to a <strong>weighted version</strong>.</p>
<p><img src="/images/notes/Cluster Management Policies-20250312123022124.webp" style="width: 100%" class="center" alt="Cluster Management Policies-20250312123022124"></p>
<h4 id="fairness-considerations">Fairness Considerations<a class="anchor" aria-hidden="true" hidden href="#fairness-considerations">#</a></h4>
<p>One can prove that this algorithm is fair:</p>
<ul>
<li>Each user gets at least <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:1em;vertical-align:-0.25em;"></span><span class="mord">1/</span><span class="mord mathnormal">n</span></span></span></span> of the resource, that is their <strong>fair share</strong>. If you demand less, you will get less, getting more is not guaranteed, it depends on the other requests.</li>
<li>Nobody gets more than what it demands</li>
<li>When a user's request is not fully satisfied, the user gets an equal share of the remaining resources</li>
<li>It is possible to extend with with priorities.</li>
<li>It is <strong>strategy proof</strong>: users are not better off if they lie about their demands.
<ul>
<li>For example a scheduler that assigns resources based on CPU utilization alone is not strategy-proof: applications could fake their usages to get more resources.</li>
</ul>
</li>
</ul>
<p>This kind of scheduling is very good, but only if you have a <strong>single resource</strong>, if you have more disaggregated resource types, then it becomes a little more difficult.</p>
<h4 id="weighted-fair-scheduling">Weighted fair scheduling<a class="anchor" aria-hidden="true" hidden href="#weighted-fair-scheduling">#</a></h4>
<p>In the weighted fair scheduling, you assign a weight to each user, and then allocate resources proportionally to the weight.
It is a little different from weighted max-min fairness. It's a good thing to do exercises to understand this weighted version of the algorithm.</p>
<h3 id="dominant-resource-fairness">Dominant Resource Fairness<a class="anchor" aria-hidden="true" hidden href="#dominant-resource-fairness">#</a></h3>
<p>This resource assignment algorithm has been introduced in <a href="#ghodsiDominantResourceFairness">(Ghodsi et al.)</a>. It is <strong>not strategy proof</strong>: some programs can make seem their dominant resource is another so that they get higher shares.</p>
<h4 id="the-idea">The Idea<a class="anchor" aria-hidden="true" hidden href="#the-idea">#</a></h4>
<p>The idea is quite simple: consider the dominant resource usage by each user. Then allocate the containers in a manner that is constant with respect to <strong>max resource usage</strong> ratio / number of containers.</p>
<p>For example, if Alice is using 6% of the available CPU, and Bob 3% of the available RAM, then bob should have twice as the number of containers compared to Alice.</p>
<h4 id="worked-out-example">Worked out Example<a class="anchor" aria-hidden="true" hidden href="#worked-out-example">#</a></h4>
<p>Suppose you have 9 CPU and 18 GB.</p>
<ul>
<li>Alice wants to issue processes 1 CPU and 4 GB</li>
<li>Bob demands to issue processes with 3CPU and 1GB.</li>
</ul>
<p>You want to equalize the share of the dominant resources with same constraints.

And we can do this for many many other kinds of tasks.</p>

<h4 id="token-bucket">Token Bucket<a class="anchor" aria-hidden="true" hidden href="#token-bucket">#</a></h4>
<ul>
<li>Has a size (number of tokens in the bucket)</li>
<li>The <strong>fill rate</strong>, how quickly it is filled up.
Commonly used for controlling network and storage traffic.
Basically the buckets are filled up at a constant rate, and spent when the transmission is needed.
The downside is that if you don't need to spend the tokens, then this values is capped</li>
</ul>
<h2 id="resource-assignment">Resource Assignment<a class="anchor" aria-hidden="true" hidden href="#resource-assignment">#</a></h2>
<p>The algorithm is easy: filter for hard constraints and order for soft constraints, this is what Kubernetes does with node affinities, you can set hard and soft constraints.
Users usually do not know what their application needs.
Often users tend to ask more than they need, this is called <strong>over-provisioning</strong>.</p>
<ul>
<li><strong>Filter</strong>: machines that satisfy <em>hard constraints</em>.</li>
<li><strong>Sort</strong>: machines that satisfy <em>soft constraints</em>, to get the best fit machine.</li>
</ul>
<h4 id="problem-of-low-resource-usage">Problem of low resource usage<a class="anchor" aria-hidden="true" hidden href="#problem-of-low-resource-usage">#</a></h4>
<p>From the point of view of the provider this is not optimal since we will have a low resource utilization:</p>
<ul>
<li>Hard consumption and costs.</li>
<li>Most of the users request more resources than they actually need.
This problem has improved (Google Borg (2020 study) cluster manager has around 50% utilization by allocating to almost double CPU!).</li>
</ul>
<h3 id="quasar-paper">Quasar Paper<a class="anchor" aria-hidden="true" hidden href="#quasar-paper">#</a></h3>
<p>This section will be wholly based on <a href="https://doi.org/10.1145/2541940.2541941">(Delimitrou &#x26; Kozyrakis 2014)</a> (Using ML to make decisions was a new thing at the time).</p>
<h4 id="performance-goal-assignments">Performance Goal assignments<a class="anchor" aria-hidden="true" hidden href="#performance-goal-assignments">#</a></h4>
<p>We want to allocate resources based on <strong>performance</strong> goals, not predicted resource usages.</p>
<ul>
<li>For each new application, we need to recommend a resource allocation and assignment</li>
<li>We need to base this reccomendation based on previous needs of similar applications.</li>
<li>It does <strong>collaborative filtering</strong> to decide the resource allocation, and adjust it <em>dynamically</em>.</li>
</ul>
<p>The idea is to use the data from previously run applications and newly run ones to get some data about QoS compliance and resource usage.</p>
<h4 id="understanding-performance-and-resource-trade-offs">Understanding Performance and Resource Trade-offs<a class="anchor" aria-hidden="true" hidden href="#understanding-performance-and-resource-trade-offs">#</a></h4>
<p>The idea is to combine information from past similar jobs from before and profiling information.
We want to be able to answer questions like: how does the new job relate to past jobs? Doing some short profiling to understand the actual resource usage to some past processes.</p>
<ul>
<li><strong>Small signal</strong>: refers to the <strong>short-term performance feedback</strong> obtained from running the process in a <em>sandboxed environment</em> to get information about resource usage and performance, and understand what kind of scaling or interference is needed.</li>
<li><strong>Large signal</strong>: refers to the <strong>past similar processes</strong> that the current run can be compared to.
The paper also asserts they do some periodic checks to ensure that the performance is still good, and adjust the resource allocation if needed, both by adding less or more resources.</li>
</ul>
<h4 id="collaborative-filtering-reccomendation">Collaborative Filtering Reccomendation<a class="anchor" aria-hidden="true" hidden href="#collaborative-filtering-reccomendation">#</a></h4>
<p><img src="/images/notes/Cluster Management Policies-20250525201026681.webp" style="width: 100%" class="center" alt="Cluster Management Policies-20250525201026681"></p>
<p>Here: the movies are the resources/platforms or interference types, and users are processes, we want to reconstruct best allocation (reconstruct missing data).
After filling out important information, they use a simple greedy scheduler to actually assign the resources.
The whole pipeline takes about 80ms (divided in three steps, profiling, classification and scheduling.)</p>
<h4 id="types-of-data-features">Types of Data Features<a class="anchor" aria-hidden="true" hidden href="#types-of-data-features">#</a></h4>
<p>The performance might be affected by the following features, Quasar takes also these variables into consideration:</p>
<ul>
<li>Scale-out (e.g. using 4 nodes vs 1 node)</li>
<li>Scale-up (e.g. using 4 CPUs vs 1 CPU)</li>
<li>Hardware heterogeneity (different CPU or memory hardware and configurations)</li>
<li>Interference (e.g. running on a shared cluster, might give you different kinds of l1i, l1d, bandwidth types of interference).</li>
</ul>
