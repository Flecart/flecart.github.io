2c:T5e36,<h3 id="little-bits-of-history">Little bits of history<a class="anchor" aria-hidden="true" hidden href="#little-bits-of-history">#</a></h3>
<p>It was invented in 1970 in Almaden (San Jose) by IBM (Don Chamberlin, Raymond Boyce worked on this) for the first relational database, called system R. Then for copyright issues it hasn't been called SEQUEL, so they branded it as SQL.</p>
<h4 id="sql-is-a-declarative-language">SQL is a declarative language<a class="anchor" aria-hidden="true" hidden href="#sql-is-a-declarative-language">#</a></h4>
<p>With declaratives language there is a separation between what I call the intentionality and the actual process. In declarative languages we just say what we want the result to be, and don't care what the actual implementation is like.
This allows queries to be executed and optimized in different ways, even if the query on the surface is the same</p>
<h4 id="sql-is-a-functional-language">SQL is a functional language<a class="anchor" aria-hidden="true" hidden href="#sql-is-a-functional-language">#</a></h4>
<p>SQL is also a functional language, because the main principle is based on <strong>expression evaluation</strong>. This allows SQL to have some nested expressions in it.</p>
<h3 id="data-types">Data types<a class="anchor" aria-hidden="true" hidden href="#data-types">#</a></h3>
<h4 id="default-data-types">Default data types<a class="anchor" aria-hidden="true" hidden href="#default-data-types">#</a></h4>
<p>I tipi di dati sono</p>
<ul>
<li>Carattere</li>
<li>numero</li>
<li>data</li>
<li>tempo</li>
<li>intervallo di tempo</li>
<li>booleano</li>
<li>blob (binario)</li>
<li>clob (carattere)</li>
</ul>
<h4 id="setting-custom-data-types">Setting custom data types<a class="anchor" aria-hidden="true" hidden href="#setting-custom-data-types">#</a></h4>
<p>Ma possono essere definiti anche tipi di dati custom, la sintassi è simile</p>
<pre><code class="language-sql">CREATE DOMAIN Grade
AS SMALLINT DEFAULT NULL
CHECK (value >= 18 AND value &#x3C;= 30)
</code></pre>
<h4 id="altering-existing-domains">Altering existing domains<a class="anchor" aria-hidden="true" hidden href="#altering-existing-domains">#</a></h4>
<p>In cui posso mettere anche dei <strong>check custom</strong>.</p>
<pre><code class="language-sql">DROP DOMAIN
</code></pre>
<p>Per cancellare il domain lì presente</p>
<p>E si può anche cambiare con</p>
<pre><code class="language-sql">ALTER DOMAIN
</code></pre>

<h4 id="inner-and-outer-joins">Inner and outer joins<a class="anchor" aria-hidden="true" hidden href="#inner-and-outer-joins">#</a></h4>

<ul>
<li><strong>Inner</strong> Dati che non hanno elementi in comune vengono scartati (come quello presente sulla slides di sopra) questo viene anche chiamato <strong>natural join</strong>.</li>
<li><strong>Outer</strong> Vengono tenuti anche i dati che non fanno matching in una parte in comune, solitamente questi sono sempre chiamati <strong>left</strong> o <strong>right</strong>, ma vedremo dopo esattamente quale sia la semantica</li>
</ul>
<p>LEFT: (right è esattamente il contrario)
Ritorna sempre il sinistro, ma il destro può anche essere null
FULL:
ritorna sinistro se c'è e destro se c'è.</p>

<h3 id="correctness-conditions">Correctness conditions<a class="anchor" aria-hidden="true" hidden href="#correctness-conditions">#</a></h3>

<h4 id="visibilita">Visibilità<a class="anchor" aria-hidden="true" hidden href="#visibilita">#</a></h4>
<p>Ci sono due note riguardo la visibilità, perché seguendo una logica simile agli <a href="/notes/nomi-e-scope/">scopes strutturali</a> se sono in scope esterno non posso accedere a quello internamente definito.
Mentre la query innestata può leggere variabili definite esternamente.

<h2 id="aggregate-functions">Aggregate functions<a class="anchor" aria-hidden="true" hidden href="#aggregate-functions">#</a></h2>
<p>Gli aggregate consentono di ritornare un valore unico da una lista di dati.
Hanno una semantica precisa:
Prima fatto tutto, ignorando dell'esistenza del groupby, fanno una selezione di tutti gli attributi che sono presenti qui, e poi effettivamente raggruppano.
Sul libro atzeni è descritto in pagina 123.
L'aspetto principale da ricordare è che <strong>attributi in group by sono superset degli attributi di selezione</strong>.</p>
<h3 id="classical-sintax">Classical sintax<a class="anchor" aria-hidden="true" hidden href="#classical-sintax">#</a></h3>
<p>La sintassi classica per questo genere di query è</p>
<pre><code class="language-pseudocode">Aggr([DISTINCT] attribute)
</code></pre>
<p>Attribute è il dominio su cui andare a runnare la funzione di aggregazione.</p>

<h3 id="grouping">Grouping<a class="anchor" aria-hidden="true" hidden href="#grouping">#</a></h3>
