21:T5477,<h3 id="perché-filesystem">Perché filesystem?<a class="anchor" aria-hidden="true" hidden href="#perché-filesystem">#</a></h3>
<p>Questa è l'idea presa dall'archivio, come se fosse un ufficio che deve tenere delle pratiche ordinate in cartelle e cartelloni.</p>
<p>L’utilizzo principale è dare un <strong>interfaccia comune di accesso ai dispositivi.</strong> perché dispositivi diversi hanno sotto modi di accedere diversi, questa interfaccia facilita molto l'accesso.</p>
<h3 id="informazioni-dei-files-5">Informazioni dei files (5+)<a class="anchor" aria-hidden="true" hidden href="#informazioni-dei-files-5">#</a></h3>
<p>Il file è <strong>l’unità logica di memorizzazione</strong>. il formato che c'è dentro è gestito dall'applicazione, non dal file system!</p>
<h4 id="lista-degli-attributi">Lista degli attributi<a class="anchor" aria-hidden="true" hidden href="#lista-degli-attributi">#</a></h4>
<pre><code>![[image/universita/ex-notion/Filesystem/Untitled.png]]

![[image/universita/ex-notion/Filesystem/Untitled 1.png]]
</code></pre>
<p>Ci sono un sacco di informazioni memorizzate nei file (un sacco di metadata anche)</p>
<ol>
<li>Nome</li>
<li>Data e ora di accesso o modifica</li>
<li>Informazioni sull'ownership e protezione di accesso.</li>
<li>Dimensione del file</li>
<li>TIpo del file</li>
</ol>
<h3 id="categorizzazione-dei-files-3">Categorizzazione dei files (3)<a class="anchor" aria-hidden="true" hidden href="#categorizzazione-dei-files-3">#</a></h3>
<ul>
<li>
<p>Tipi di files</p>
<p><img src="/images/notes/image/universita/ex-notion/Filesystem/Untitled 2.png" style="width: 100%" class="center" alt="Untitled 2"></p>
</li>
</ul>
<p>Possiamo andare a distinguere i files a seconda del</p>
<ul>
<li><strong>Contenuto</strong> (se contiene codice oggetto, se contiene dati o informazioni)</li>
<li><strong>Struttura</strong>, se sono plain text sequenze di bytes, oppure dati strutturati come database., e possono anche essere un albero**.**</li>
<li>UNIX: files speciali (blocchi o caratteri), pipes, oltre ai classici files e directories. NOTA: ci sono molte cose che sono viste come files, in unix tutto è un file :D.
<ul>
<li>Sono identificati da una coppia di numeri che identificano il device driver</li>
</ul>
</li>
</ul>
<h3 id="files-in-sistemi-operativi-comuni">Files in sistemi operativi comuni<a class="anchor" aria-hidden="true" hidden href="#files-in-sistemi-operativi-comuni">#</a></h3>
<p>I files sono nati con la metafora dell’ufficio:</p>
<ul>
<li>Archivi di informazione, che vanno aperti e richiusi (messi nella loro collocazione corretta)</li>
</ul>
<p>È utile conoscere il tipo di file per capire quale utilizzo andiamo a fare di essa.</p>
<p>C’è un tradeoff, se ho tanti tipi di files, il sistema operativo diventa molto più complesso, se ho troppi pochi potrebbero essere troppo generali.</p>
<ul>
<li>
<p>Slide tradeoff tipologie di files</p>
<p><img src="/images/notes/image/universita/ex-notion/Filesystem/Untitled 3.png" style="width: 100%" class="center" alt="Untitled 3"></p>
</li>
</ul>
<h3 id="tecniche-di-identificazione-dei-file-3">Tecniche di identificazione dei file (3)<a class="anchor" aria-hidden="true" hidden href="#tecniche-di-identificazione-dei-file-3">#</a></h3>
<p><img src="/images/notes/image/universita/ex-notion/Filesystem/Untitled 4.png" style="width: 100%" class="center" alt="Untitled 4"></p>
<p>Generalmente ci sono 3 modi per riconoscere il tipo di files:</p>
<ul>
<li>Magic numbers</li>
<li>Attributo tipo del filesystem</li>
<li>estensione.</li>
</ul>
<p>Rispettivamente queste tecniche sono utilizzate in:</p>
<ol>
<li>Win (estensioni)</li>
<li>Mac, informazioni aggiuntive sui files</li>
<li>Unix, il magic number all’inizio dei files</li>
</ol>
<ul>
<li>
<p>Slide metodi di riconoscimento nei sistemi di riconoscimento</p>
<p><img src="/images/notes/image/universita/ex-notion/Filesystem/Untitled 5.png" style="width: 100%" class="center" alt="Untitled 5"></p>
</li>
</ul>
<p>Unix riconosce solamente gli eseguibili, in questo senso è minimalista.</p>
<p>Col magic number riesce anche a <strong>riconoscere l’architettura target dell’eseguibile</strong>.</p>
<p>Mac OS ha un codice identificativo per un programma che ha creato il file o la risorsa.</p>
<p>Windows guarda l'estensione. che solitamente sempre da 3 caratteri. (inizilamente il nome del file aveva solo 8 caratteri.</p>

<h3 id="semantica-di-coerenza">Semantica di coerenza<a class="anchor" aria-hidden="true" hidden href="#semantica-di-coerenza">#</a></h3>
<p>Dato che i sistemi moderni sono maggiormente tutti multitasking, vogliamo andare a specificare quando una modifica di un file può essere vista da un altro processo.</p>
<p><strong>immediato:</strong> questa è la semantica che viene utilizzata anche nei sistemi UNIX, una modifica è subito vista da altri programmi. Bisogna chiedersi ora come si faccia ad implementare una cosa di  questo genere.</p>
<p><strong>AFS</strong>: è un esempio di file system interplanetario, in cui non era possibile fare una semantica imemdiata (i dati erano sparsi in mezzo al mondo). <strong>semantica di coerenza delle sessioni</strong> è il nome di questo. ossia il file veniva modificato quando il file era chiuso. Questo era necessario perché ogni write dovevano rendere traffico sulla rete, era troppa questa sincronizzazione e troppo traffico direi.</p>
