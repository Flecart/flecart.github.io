30:T47c4,<blockquote>
<p>"I call it my billion-dollar mistake. It was the invention of the null reference in 1965. [...] This has led to innumerable errors, vulnerabilities, and system crashes." ~ <em>Tony Hoare</em> 1</p>
</blockquote>
<p>The core problem addressed here is <strong>Null Safety</strong> and how to properly initialize objects to ensure type invariants (specifically non-nullness) hold true.</p>
<h3 id="simple-non-null-types">Simple Non-Null Types<a class="anchor" aria-hidden="true" hidden href="#simple-non-null-types">#</a></h3>
<p>The goal is to prevent null-dereferencing errors statically.
Most of the variables are null, after initialization.</p>
<h4 id="non-null-types-vs-possibly-null-types">Non-Null Types vs. Possibly-Null Types<a class="anchor" aria-hidden="true" hidden href="#non-null-types-vs-possibly-null-types">#</a></h4>
<ul>
<li><strong>Non-null type <code>T!</code></strong>: Consists of references to <code>T</code>-objects. It <em>cannot</em> hold null.</li>
<li><strong>Possibly-null type <code>T?</code></strong>: Consists of references to <code>T</code>-objects plus <code>null</code>. This corresponds to the standard type <code>T</code> in languages like Java, and the most common case among most languages.</li>
</ul>
<!--SR:!2025-12-21,12,270-->
<h4 id="type-safety-and-invariant">Type Safety and Invariant<a class="anchor" aria-hidden="true" hidden href="#type-safety-and-invariant">#</a></h4>
<ul>
<li><strong>Invariant</strong>: If the static type of an expression <code>e</code> is a non-null type, then <code>e</code>'s value at runtime must be different from <code>null</code>.</li>
<li><strong>Enforcement</strong>: We require non-null types for the receiver of each field access, array access, method call, and throw statement. Equivalent to message not understood cited in <a href="/notes/typing-and-subtyping/">Typing and Subtyping</a>.</li>
</ul>
<h4 id="subtyping-and-casts">Subtyping and Casts<a class="anchor" aria-hidden="true" hidden href="#subtyping-and-casts">#</a></h4>
<p>The values of T! are a proper subset of T?.</p>
<ul>
<li><strong>Subtyping</strong>:
<ul>
<li><code>S! &#x3C;: T!</code> (if S extends T)</li>
<li><code>S? &#x3C;: T?</code> (if S extends T)</li>
<li><code>T! &#x3C;: T?</code> (Non-null is a subtype of possibly-null) 6.</li>
</ul>
</li>
<li><strong>Downcasts</strong>: Casting from <code>T?</code> (possibly-null) to <code>T!</code> (non-null) requires a runtime check., so we know <code>T?</code> is not a subtype of <code>T!</code>.</li>
</ul>

<h4 id="type-rules">Type Rules<a class="anchor" aria-hidden="true" hidden href="#type-rules">#</a></h4>
<p>Some expressions in a language <strong>need to be non-null</strong></p>
<ul>
<li>Receiver of field access</li>
<li>Receiver of array access</li>
<li>Receiver of method call</li>
<li>Expression of a throw statement.
If not, you can just get a null exception. Now we would like to have compile time error, to make things statically safe.</li>
</ul>
<!--SR:!2025-12-11,2,230-->
<h4 id="safe-handling-null">Safe Handling Null<a class="anchor" aria-hidden="true" hidden href="#safe-handling-null">#</a></h4>
<p>Control Flow Analysis (Dataflow Analysis)
How do we safely use a possibly-null type?
We can check against null.</p>
<ul>
<li><strong>Definite Assignment</strong>: Java/C# use dataflow analysis to ensure local variables are assigned before use.</li>
<li><strong>Null Checks</strong>: If we check <code>if (n != null)</code>, the compiler can treat <code>n</code> as non-null inside the block.</li>
<li><strong>Limitations</strong>: Dataflow analysis works well for local variables but <em>not</em> for heap locations (fields) because of aliasing and <em>side</em> effects (e.g., a method call <code>foo(this)</code> might set a field to null behind the scenes). <em>Concurrency might be a problem</em>.
I have personally seen this thing for the first time in Typescript during my internship at Cubbit.</li>
</ul>
<!--SR:!2025-12-18,9,250-->
<h3 id="object-initialization">Object Initialization<a class="anchor" aria-hidden="true" hidden href="#object-initialization">#</a></h3>
<p>The main challenge is: How do we construct an object of a non-null type?</p>
<p>When we create an object (new T()), fields start as null (or default values). We must ensure all non-null fields are assigned non-null values before the constructor terminates.</p>
<p>The purpose is <strong>establishing</strong> some invariant. We want to know <em>when</em> it is possible to rely on the invariants.</p>
<h4 id="the-escaping-problem">The Escaping Problem<a class="anchor" aria-hidden="true" hidden href="#the-escaping-problem">#</a></h4>
<p>A naive definite assignment check for fields isn't enough. If the this reference "escapes" the constructor (is passed to another method, stored in a global field, etc.) before the object is fully initialized, other code might see null fields that are supposed to be T!.</p>
<p><strong>Escaping Scenarios</strong>:</p>
<ol>
<li><strong>Method Calls</strong>: Calling a dynamically bound method on <code>this</code> inside the constructor. A subclass might override this method and access a field that hasn't been initialized yet.</li>
<li><strong>Callbacks</strong>: Registering <code>this</code> as a listener (e.g., Observer pattern) inside the constructor. The Subject might call back immediately.</li>
<li><strong>Concurrent Access</strong>: Publishing <code>this</code> to a static field where another thread picks it up, bu the object has not been initialized yet.</li>
</ol>
<!--SR:!2025-12-11,2,230-->
<h4 id="initialization-phases">Initialization Phases<a class="anchor" aria-hidden="true" hidden href="#initialization-phases">#</a></h4>
<p>To solve this, we track the state of the object during construction using a type system.</p>
<ol>
<li><strong>Free (Under Construction)</strong>: The object is being created. Fields may be null.</li>
<li><strong>Committed (Initialized)</strong>: The object is fully initialized. All non-null invariants hold.</li>
</ol>
<!--SR:!2025-12-20,11,270-->
<h4 id="construction-types">Construction Types<a class="anchor" aria-hidden="true" hidden href="#construction-types">#</a></h4>
<p>For every class T, we distinguish three states for references:</p>
<ul>
<li><strong><code>T!</code> / <code>T?</code> (Committed)</strong>: Standard types. Construction is complete, if you read non null, it means, we only read non null, for the other cases its not guaranteed.</li>
<li><strong><code>free T!</code> / <code>free T?</code> (Free)</strong>: References to objects <em>under construction</em> (like <code>this</code> inside a constructor).</li>
<li><strong><code>unc T!</code> / <code>unc T?</code> (Unclassified)</strong>: We don't know if it's free or committed (common supertype).</li>
<li>There are no casts from unclassified to free or committed types.</li>
</ul>
<!--SR:!2025-12-20,11,270-->
<h4 id="initialization-requirements">Initialization Requirements<a class="anchor" aria-hidden="true" hidden href="#initialization-requirements">#</a></h4>
<ul>
<li><strong>Requirement 1: Local Initialization</strong>: An object is locally initialized if its non-null fields have non-null values. Committed types <em>must</em> be locally initialized.</li>
<li><strong>Requirement 2: Transitive Initialization</strong>: If an object is committed, everything it reaches (references) must also be committed (transitively initialized).</li>
</ul>

<h4 id="handling-cyclic-structures">Handling Cyclic Structures<a class="anchor" aria-hidden="true" hidden href="#handling-cyclic-structures">#</a></h4>
<p>Cyclic structures (e.g., a Node pointing to another Node) pose a problem. We cannot have both objects fully "committed" before they reference each other.</p>
<ul>
<li><em>Solution</em>: We allow <code>free</code> references to be assigned to fields of an object under construction.</li>
<li>The constructor parameters can be declared <code>free</code> to allow passing partially initialized objects (like <code>this</code>).
<ul>
<li>It will be completed only after the last one is completed for requirement two of being well typed.</li>
</ul>
</li>
</ul>
<pre><code class="language-java">
class List {
    List! next; // cyclic list

    List(int n) {
        if (n == 1) {
            next = this;
        } else {
            next = new List(this, n);
        }
    }

    List(free List! last, int n) {
        if (n == 2) {
            next = last;
        } else {
            next = new List(last, n - 1);

<h3 id="initialization-of-global-data">Initialization of Global Data<a class="anchor" aria-hidden="true" hidden href="#initialization-of-global-data">#</a></h3>
<p>Global data (Singletons, Factories, Flyweights) must be initialized before access.</p>
<h4 id="design-goals">Design Goals<a class="anchor" aria-hidden="true" hidden href="#design-goals">#</a></h4>

<li><strong>Laziness</strong>: Initialize only when needed to save startup time.</li>

<h4 id="global-vars-and-init-methods">Global Vars and Init-Methods<a class="anchor" aria-hidden="true" hidden href="#global-vars-and-init-methods">#</a></h4>
<p>This approach uses global variables to store references to global data, but relies on explicit calls to initialization methods to set them up. This is often the most basic way to handle globals in languages that support them.</p>
<ul>
<li><strong>Mechanism</strong>: Explicit <code>init()</code> calls that must be invoked, usually from a main function.</li>
<li><strong>Pros</strong>: It is simple to implement.</li>
<li><strong>Cons</strong>:
<ul>
<li><strong>Manual Ordering</strong>: The programmer must manually code the order of initialization to satisfy dependencies, which is error-prone.</li>
<li><strong>Encapsulation</strong>: Main methods often need to know internal module dependencies to call inits in the right order, breaking information hiding.</li>
<li><strong>No Laziness</strong>: It generally requires upfront initialization unless manually coded otherwise.</li>
</ul>
</li>
</ul>

global Factory theFactory;

    theFactory = new Factory( );

class Factory {
    HashMap flyweights;
    Flyweight create( Data d ) { ... }

<h4 id="static-fields-and-initializers-java/c">Static Fields and Initializers (Java/C#)<a class="anchor" aria-hidden="true" hidden href="#static-fields-and-initializers-java/c">#</a></h4>
<p>Java and C# use static fields to store global data and static initializer blocks to initialize them. These blocks run automatically immediately before the class is first used (e.g., creation of an instance, static method call, or static field access)666.</p>
<ul>
<li><strong>Mechanism</strong>: <code>static { ... }</code> blocks executed by the runtime system.</li>
<li><strong>Pros</strong>:
<ul>
<li><strong>Automatic &#x26; Lazy</strong>: Initialization happens just in time when the class is needed, reducing startup time7.</li>
<li><strong>System-managed</strong>: The system handles the triggering of initialization8.</li>
</ul>
</li>
<li><strong>Cons</strong>:
<ul>
<li><strong>Mutual Dependencies</strong>: If class A's static initializer triggers class B, and class B triggers class A, the cycle can lead to crashes or <code>NullPointerExceptions</code> because initialization is considered "in progress" and won't restart, leaving fields uninitialized 9.</li>
<li><strong>Side Effects</strong>: Static initializers can have arbitrary side effects (like modifying other static fields), making it hard to reason about the program state since execution order depends on which class is accessed first 10.</li>
</ul>
</li>
</ul>
<pre><code class="language-java">class Factory {
    static Factory theFactory;
    HashMap flyweights;

    // Static initializer block
    static {
        theFactory = new Factory( );
    }

    Flyweight create( Data d ) { ... }
}

// Initialization triggered here automatically
Factory o = Factory.theFactory;
Flyweight f = o.create( ... );
</code></pre>

<h4 id="eiffel-once-methods">Eiffel Once Methods<a class="anchor" aria-hidden="true" hidden href="#eiffel-once-methods">#</a></h4>
<p>Eiffel uses <code>once</code> methods (routines). The body of a <code>once</code> method is executed only the first time it is called. The result is cached and returned for all subsequent calls.</p>
<ul>
<li><strong>Mechanism</strong>: The <code>once</code> keyword applied to a feature (method).</li>
<li><strong>Pros</strong>:
<ul>
<li><strong>Laziness</strong>: The initialization code runs only when the data is actually requested.</li>
<li><strong>Caching</strong>: Provides a consistent global access point.</li>

<li><strong>Cons</strong>:

<li><strong>Recursion/Mutual Dependencies</strong>: If a <code>once</code> method recursively calls itself (directly or via another object) during its first execution, it returns the <em>current</em> (partial) result rather than waiting or crashing. </ul>
