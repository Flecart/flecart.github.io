2b:Tf43c,<h2 id="universal-composability-uc-framework">Universal Composability (UC) Framework<a class="anchor" aria-hidden="true" hidden href="#universal-composability-uc-framework">#</a></h2>
<h3 id="motivation-the-composition-problem">Motivation: The Composition Problem<a class="anchor" aria-hidden="true" hidden href="#motivation-the-composition-problem">#</a></h3>
<p>The fundamental issue UC addresses: protocols proven secure in isolation can fail catastrophically when run concurrently with other protocols. Classical security definitions (e.g., standalone simulation-based security) do not guarantee that security properties are preserved under arbitrary composition. Real-world systems always run multiple protocols simultaneously, sharing state, keys, randomness, and communication channels.</p>

<p>Key failure modes under composition:</p>
<ul>
<li><strong>Shared state exploitation</strong>: Protocol A leaks information that helps an adversary attack Protocol B.</li>
<li><strong>Interleaving attacks</strong>: An adversary schedules messages across concurrent sessions to create attack vectors absent in any single session.</li>
<li><strong>Environmental interference</strong>: External protocols provide the adversary with auxiliary inputs or side channels not modeled in standalone analysis.</li>
</ul>
<p>The UC framework, introduced by <strong>Ran Canetti (2001)</strong>, provides a security definition that is <em>preserved under arbitrary composition by construction</em>.</p>
<p>See Simulation-Based Security, Concurrent Composition, MPC</p>
<h3 id="the-uc-framework-architecture">The UC Framework: Architecture<a class="anchor" aria-hidden="true" hidden href="#the-uc-framework-architecture">#</a></h3>
<h4 id="the-execution-model">The Execution Model<a class="anchor" aria-hidden="true" hidden href="#the-execution-model">#</a></h4>
<p>The UC model defines security via an <strong>interaction among four entities</strong>:</p>

























<table><thead><tr><th align="left">Entity</th><th align="left">Role</th></tr></thead><tbody><tr><td align="left"><strong>Environment <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0794em;">Z</span></span></span></span></strong></td><td align="left">Controls all inputs to and reads all outputs from parties; represents <em>everything external</em> to the protocol, including other concurrent protocols</td></tr><tr><td align="left"><strong>Adversary <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal">A</span></span></span></span></strong></td><td align="left">Attacks the real protocol (corrupts parties, controls network)</td></tr><tr><td align="left"><strong>Simulator <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.075em;">S</span></span></span></span></strong></td><td align="left">Simulates the real-world view in the ideal world; must fool <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0794em;">Z</span></span></span></span></td></tr><tr><td align="left"><strong>Ideal Functionality <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0993em;">F</span></span></span></span></strong></td><td align="left">Trusted third party specifying <em>what</em> the protocol should compute</td></tr></tbody></table>
<p>The critical distinction from classical simulation: <strong><span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0794em;">Z</span></span></span></span> is interactive and adaptive</strong>. It provides inputs, receives outputs, and can interact with the adversary <em>during execution</em>. This models arbitrary concurrent context.</p>
<h4 id="ideal-functionality-mathcalf">Ideal Functionality <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0993em;">F</span></span></span></span><a class="anchor" aria-hidden="true" hidden href="#ideal-functionality-mathcalf">#</a></h4>

<h4 id="uc-security-definition">UC-Security Definition<a class="anchor" aria-hidden="true" hidden href="#uc-security-definition">#</a></h4>
<blockquote>

<h3 id="the-uc-composition-theorem">The UC Composition Theorem<a class="anchor" aria-hidden="true" hidden href="#the-uc-composition-theorem">#</a></h3>
<h4 id="statement-of-the-theorem">Statement of the Theorem<a class="anchor" aria-hidden="true" hidden href="#statement-of-the-theorem">#</a></h4>
<blockquote>
<p><strong>UC Composition Theorem (Canetti 2001):</strong> If protocol <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.4306em;"></span><span class="mord mathnormal" style="margin-right:0.0359em;">π</span></span></span></span> UC-realizes <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0993em;">F</span></span></span></span>, then for any protocol <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.625em;vertical-align:-0.1944em;"></span><span class="mord mathnormal">ρ</span></span></span></span> that uses <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0993em;">F</span></span></span></span> as a subroutine (i.e., makes "ideal calls" to <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0993em;">F</span></span></span></span>), the composed protocol <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:1.0824em;vertical-align:-0.1944em;"></span><span class="mord"><span class="mord mathnormal">ρ</span><span class="msupsub"><span class="vlist-t"><span class="vlist-r"><span class="vlist" style="height:0.888em;"><span style="top:-3.063em;margin-right:0.05em;"><span class="pstrut" style="height:2.7em;"></span><span class="sizing reset-size6 size3 mtight"><span class="mord mtight"><span class="mord mathnormal mtight" style="margin-right:0.0359em;">π</span><span class="mord mtight">/</span><span class="mord mathcal mtight" style="margin-right:0.0993em;">F</span></span></span></span></span></span></span></span></span></span></span></span> — obtained by replacing each call to <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0993em;">F</span></span></span></span> with an execution of <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.4306em;"></span><span class="mord mathnormal" style="margin-right:0.0359em;">π</span></span></span></span> — UC-realizes whatever <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.625em;vertical-align:-0.1944em;"></span><span class="mord mathnormal">ρ</span></span></span></span> UC-realizes.</p>
</blockquote>
<p>This is the <em>entire point</em> of the framework. It enables <strong>modular protocol design</strong>:</p>
<ol>
<li>Design and analyze protocol components against ideal functionalities.</li>
<li>Compose them freely — security is guaranteed to hold.</li>
</ol>
<p>The theorem holds for:</p>
<ul>
<li><strong>Self-composition</strong>: Multiple concurrent sessions of the same protocol.</li>
<li><strong>General composition</strong>: Arbitrary concurrent protocols, including those designed adversarially.</li>
</ul>
<h4 id="why-the-environment-makes-this-work">Why the Environment Makes This Work<a class="anchor" aria-hidden="true" hidden href="#why-the-environment-makes-this-work">#</a></h4>
<p>In standalone simulation, the distinguisher sees only the protocol's output. In UC, <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0794em;">Z</span></span></span></span> models the entire external context:</p>
<ul>
<li>It can run other protocols concurrently.</li>
<li>It can feed outputs of one protocol as inputs to another.</li>
<li>It can adaptively choose inputs based on observed behavior.</li>
</ul>
<p>Because UC-security holds against <em>every</em> such <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0794em;">Z</span></span></span></span>, any composition scenario is already captured. The composition theorem then follows almost "for free" from the definition.</p>
<h3 id="setup-assumptions-and-impossibility-results">Setup Assumptions and Impossibility Results<a class="anchor" aria-hidden="true" hidden href="#setup-assumptions-and-impossibility-results">#</a></h3>
<h4 id="uc-impossibility-without-setup">UC Impossibility Without Setup<a class="anchor" aria-hidden="true" hidden href="#uc-impossibility-without-setup">#</a></h4>
<blockquote>
<p><strong>Canetti-Fischlin (2001):</strong> UC-secure commitment (and hence UC-secure OT, MPC for general functions, zero-knowledge for NP) is <strong>impossible</strong> in the plain model (no setup) against adaptive, static, or even non-adaptive adversaries corrupting one party.</p>
</blockquote>
<p>This is a fundamental negative result. The intuition: without setup, the simulator cannot "extract" the adversary's input (for binding) while simultaneously "equivocating" (for hiding), because <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0794em;">Z</span></span></span></span> can check consistency across executions.</p>
<h4 id="setup-assumptions">Setup Assumptions<a class="anchor" aria-hidden="true" hidden href="#setup-assumptions">#</a></h4>
<p>To circumvent impossibility, UC protocols rely on <strong>trusted setup</strong>:</p>



































<table><thead><tr><th align="left">Setup Model</th><th align="left">Description</th><th align="left">UC-Feasibility</th></tr></thead><tbody><tr><td align="left"><strong>Common Reference String (CRS)</strong></td><td align="left">Trusted random string available to all parties</td><td align="left">UC-MPC for any functionality (Canetti-Lindell-Ostrovsky-Sahai 2002)</td></tr><tr><td align="left"><strong>Random Oracle (RO)</strong></td><td align="left">Ideal hash function accessible by all</td><td align="left">UC-commitment, UC-OT achievable</td></tr><tr><td align="left"><strong>Public-Key Infrastructure (PKI)</strong></td><td align="left">Trusted registration of public keys</td><td align="left">Enables UC-secure protocols</td></tr><tr><td align="left"><strong>Tamper-Proof Hardware</strong> (e.g., smart cards)</td><td align="left">Parties can create and exchange hardware tokens</td><td align="left">UC without CRS (Katz 2007)</td></tr><tr><td align="left"><strong>Global Setup</strong> (<span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.7805em;vertical-align:-0.0972em;"></span><span class="mord mathcal" style="margin-right:0.0593em;">G</span></span></span></span>)</td><td align="left">Shared functionality across all sessions (e.g., global RO, global CRS)</td><td align="left">Canetti-Dodis-Pass-Walfish (2007): requires careful modeling</td></tr></tbody></table>
<p><strong>Warning</strong>: The CRS model implicitly assumes a trusted party for setup. This is sometimes criticized as "kicking the can" — you assumed away the trust problem at the setup phase. Global setup (<span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.9694em;vertical-align:-0.2861em;"></span><span class="mord"><span class="mord mathcal" style="margin-right:0.0593em;">G</span><span class="msupsub"><span class="vlist-t vlist-t2"><span class="vlist-r"><span class="vlist" style="height:0.3283em;"><span style="top:-2.55em;margin-left:-0.0593em;margin-right:0.05em;"><span class="pstrut" style="height:2.7em;"></span><span class="sizing reset-size6 size3 mtight"><span class="mord mtight"><span class="mord text mtight"><span class="mord mtight">gRO</span></span></span></span></span></span><span class="vlist-s">​</span></span><span class="vlist-r"><span class="vlist" style="height:0.2861em;"><span></span></span></span></span></span></span></span></span></span>, <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.9694em;vertical-align:-0.2861em;"></span><span class="mord"><span class="mord mathcal" style="margin-right:0.0593em;">G</span><span class="msupsub"><span class="vlist-t vlist-t2"><span class="vlist-r"><span class="vlist" style="height:0.3283em;"><span style="top:-2.55em;margin-left:-0.0593em;margin-right:0.05em;"><span class="pstrut" style="height:2.7em;"></span><span class="sizing reset-size6 size3 mtight"><span class="mord mtight"><span class="mord text mtight"><span class="mord mtight">gCRS</span></span></span></span></span></span><span class="vlist-s">​</span></span><span class="vlist-r"><span class="vlist" style="height:0.2861em;"><span></span></span></span></span></span></span></span></span></span>) addresses the concern that the CRS should be shared across <em>all</em> protocols, not per-session.</p>
<p>See Common Reference String, Random Oracle Model, Oblivious Transfer</p>
<h3 id="modeling-choices-and-variants">Modeling Choices and Variants<a class="anchor" aria-hidden="true" hidden href="#modeling-choices-and-variants">#</a></h3>
<h4 id="static-vs-adaptive-corruption">Static vs. Adaptive Corruption<a class="anchor" aria-hidden="true" hidden href="#static-vs-adaptive-corruption">#</a></h4>




















<table><thead><tr><th align="left">Model</th><th align="left">Adversary Power</th><th align="left">Difficulty</th></tr></thead><tbody><tr><td align="left"><strong>Static</strong></td><td align="left">Corrupts parties before execution begins</td><td align="left">Easier to achieve</td></tr><tr><td align="left"><strong>Adaptive</strong></td><td align="left">Corrupts parties during execution based on transcript</td><td align="left">Much harder; simulator must retroactively explain corrupted party's state</td></tr></tbody></table>
<p>Adaptive UC-security often requires <strong>equivocable</strong> and <strong>extractable</strong> primitives simultaneously, which is why many UC protocols are only proven under static corruption.</p>
<h4 id="synchronous-vs-asynchronous-communication">Synchronous vs. Asynchronous Communication<a class="anchor" aria-hidden="true" hidden href="#synchronous-vs-asynchronous-communication">#</a></h4>
<ul>
<li><strong>Synchronous</strong>: Rounds are well-defined; messages arrive within known bounds. Simplifies protocol design.</li>
<li><strong>Asynchronous</strong>: No timing guarantees; adversary controls message delivery order and delay. More realistic but harder to achieve.</li>
</ul>
<p>UC can model both. The adversary controls the network, so asynchronous is the default. Synchronous models are captured by adding a "clock" functionality <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.8333em;vertical-align:-0.15em;"></span><span class="mord"><span class="mord mathcal" style="margin-right:0.0993em;">F</span><span class="msupsub"><span class="vlist-t vlist-t2"><span class="vlist-r"><span class="vlist" style="height:0.3361em;"><span style="top:-2.55em;margin-left:-0.0993em;margin-right:0.05em;"><span class="pstrut" style="height:2.7em;"></span><span class="sizing reset-size6 size3 mtight"><span class="mord mtight"><span class="mord text mtight"><span class="mord mtight">clock</span></span></span></span></span></span><span class="vlist-s">​</span></span><span class="vlist-r"><span class="vlist" style="height:0.15em;"><span></span></span></span></span></span></span></span></span></span>.</p>
<h4 id="the-dummy-adversary-lemma">The Dummy Adversary Lemma<a class="anchor" aria-hidden="true" hidden href="#the-dummy-adversary-lemma">#</a></h4>
<blockquote>
<p>It suffices to prove UC-security against the <strong>dummy adversary</strong> <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0278em;">D</span></span></span></span>, who simply forwards all communication to <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0794em;">Z</span></span></span></span>. If the protocol is secure against <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0278em;">D</span></span></span></span>, it is secure against all PPT adversaries.</p>
</blockquote>
<p>This simplifies proofs: instead of quantifying over all <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal">A</span></span></span></span>, you fix <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal">A</span><span class="mspace" style="margin-right:0.2778em;"></span><span class="mrel">=</span><span class="mspace" style="margin-right:0.2778em;"></span></span><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0278em;">D</span></span></span></span> and absorb its power into <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0794em;">Z</span></span></span></span>. The resulting game is: <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0794em;">Z</span></span></span></span> directly controls corrupted parties and must be unable to distinguish real from ideal.</p>
<h3 id="key-uc-realized-functionalities">Key UC-Realized Functionalities<a class="anchor" aria-hidden="true" hidden href="#key-uc-realized-functionalities">#</a></h3>
<h4 id="uc-commitment-mathcalf-textcom">UC Commitment (<span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.8333em;vertical-align:-0.15em;"></span><span class="mord"><span class="mord mathcal" style="margin-right:0.0993em;">F</span><span class="msupsub"><span class="vlist-t vlist-t2"><span class="vlist-r"><span class="vlist" style="height:0.3283em;"><span style="top:-2.55em;margin-left:-0.0993em;margin-right:0.05em;"><span class="pstrut" style="height:2.7em;"></span><span class="sizing reset-size6 size3 mtight"><span class="mord mtight"><span class="mord text mtight"><span class="mord mtight">COM</span></span></span></span></span></span><span class="vlist-s">​</span></span><span class="vlist-r"><span class="vlist" style="height:0.15em;"><span></span></span></span></span></span></span></span></span></span>)<a class="anchor" aria-hidden="true" hidden href="#uc-commitment-mathcalf-textcom">#</a></h4>

<ol>
<li><strong>Commit phase</strong>: Committer sends <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:1em;vertical-align:-0.25em;"></span><span class="mopen">(</span><span class="mord mathnormal">co</span><span class="mord mathnormal">mmi</span><span class="mord mathnormal">t</span><span class="mpunct">,</span><span class="mspace" style="margin-right:0.1667em;"></span><span class="mord mathnormal">s</span><span class="mord mathnormal">i</span><span class="mord mathnormal">d</span><span class="mpunct">,</span><span class="mspace" style="margin-right:0.1667em;"></span><span class="mord mathnormal" style="margin-right:0.0359em;">v</span><span class="mclose">)</span></span></span></span> to <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0993em;">F</span></span></span></span>. <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0993em;">F</span></span></span></span> stores <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.4306em;"></span><span class="mord mathnormal" style="margin-right:0.0359em;">v</span></span></span></span>, notifies receiver of commitment (not <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.4306em;"></span><span class="mord mathnormal" style="margin-right:0.0359em;">v</span></span></span></span>).</li>

<p>UC-secure commitment requires both:</p>
<ul>

<p><span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.8333em;vertical-align:-0.15em;"></span><span class="mord"><span class="mord mathcal" style="margin-right:0.0993em;">F</span><span class="msupsub"><span class="vlist-t vlist-t2"><span class="vlist-r"><span class="vlist" style="height:0.3283em;"><span style="top:-2.55em;margin-left:-0.0993em;margin-right:0.05em;"><span class="pstrut" style="height:2.7em;"></span><span class="sizing reset-size6 size3 mtight"><span class="mord mtight"><span class="mord text mtight"><span class="mord mtight">ZK</span></span></span></span></span></span><span class="vlist-s">​</span></span><span class="vlist-r"><span class="vlist" style="height:0.15em;"><span></span></span></span></span></span></span></span></span></span> for relation <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathnormal" style="margin-right:0.0077em;">R</span></span></span></span>:</p>
<ol>

<h4 id="uc-secure-multi-party-computation">UC Secure Multi-Party Computation<a class="anchor" aria-hidden="true" hidden href="#uc-secure-multi-party-computation">#</a></h4>
<blockquote>
<p><strong>Canetti-Lindell-Ostrovsky-Sahai (2002):</strong> In the <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.8333em;vertical-align:-0.15em;"></span><span class="mord"><span class="mord mathcal" style="margin-right:0.0993em;">F</span><span class="msupsub"><span class="vlist-t vlist-t2"><span class="vlist-r"><span class="vlist" style="height:0.3283em;"><span style="top:-2.55em;margin-left:-0.0993em;margin-right:0.05em;"><span class="pstrut" style="height:2.7em;"></span><span class="sizing reset-size6 size3 mtight"><span class="mord mtight"><span class="mord text mtight"><span class="mord mtight">CRS</span></span></span></span></span></span><span class="vlist-s">​</span></span><span class="vlist-r"><span class="vlist" style="height:0.15em;"><span></span></span></span></span></span></span></span></span></span>-hybrid model, every well-formed functionality can be UC-realized assuming enhanced trapdoor permutations, tolerating any number of static corruptions.</p>
</blockquote>
<p>This is the grand feasibility result. The protocol is typically:</p>
<ol>
<li>UC-commit inputs.</li>
<li>UC-ZK prove consistency.</li>
<li>Run a garbled circuit / GMW-style protocol.</li>
</ol>
<p>See Garbled Circuits, GMW Protocol, Oblivious Transfer</p>
<h3 id="practical-considerations-and-criticisms">Practical Considerations and Criticisms<a class="anchor" aria-hidden="true" hidden href="#practical-considerations-and-criticisms">#</a></h3>
<h4 id="drawbacks-and-limitations">Drawbacks and Limitations<a class="anchor" aria-hidden="true" hidden href="#drawbacks-and-limitations">#</a></h4>
<ul>
<li><strong>Efficiency</strong>: UC-secure protocols are typically much less efficient than standalone-secure alternatives, due to the need for extractable/equivocable primitives and CRS-based constructions.</li>
<li><strong>Setup trust</strong>: Reliance on CRS/PKI/hardware tokens shifts the trust problem rather than eliminating it.</li>
<li><strong>Over-conservatism</strong>: UC-security may be <em>too strong</em> for some applications. If you know the composition context, weaker notions (e.g., <strong>Angel-based security</strong>, <strong>Generalized UC (GUC)</strong>) may suffice with better efficiency.</li>
<li><strong>Modeling complexity</strong>: Writing ideal functionalities correctly is non-trivial. Subtle differences in <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathcal" style="margin-right:0.0993em;">F</span></span></span></span> specification can lead to very different security guarantees.</li>
<li><strong>Adaptive corruption gap</strong>: Many protocols only achieve static-corruption UC-security; adaptive security remains open or requires strong assumptions (e.g., non-committing encryption).</li>
</ul>
<h4 id="uc-vs-standalone-security">UC vs. Standalone Security<a class="anchor" aria-hidden="true" hidden href="#uc-vs-standalone-security">#</a></h4>

<h4 id="variants-and-extensions">Variants and Extensions<a class="anchor" aria-hidden="true" hidden href="#variants-and-extensions">#</a></h4>
