21:T3bb0,<p>How can we coordinate services to actually understand what they are doing, or what the user wants them to do? How to manage networks errors? This note will mainly focus on high level communication protocols to coordinate this kind of communication.</p>
<h2 id="remote-procedure-calls">Remote Procedure Calls<a class="anchor" aria-hidden="true" hidden href="#remote-procedure-calls">#</a></h2>
<h3 id="history-the-stub">History: the Stub<a class="anchor" aria-hidden="true" hidden href="#history-the-stub">#</a></h3>
<p>This has been the main idea, introduced in 1984, using the idea of <strong>stubs</strong>, see <a href="https://dl.acm.org/doi/10.1145/2080.357392">(Birrell &#x26; Nelson 1984)</a>. The system basically calls the remote procedure as i<em>f it was local</em> on the high level, but on a lower level a network request is sent.
The architecture has remained the same in these years.
It hides all the complexity in the stub (marshaling, binding and sending, without caring about the sockets and communication matters).
One problem is that it might be <em>hiding</em> the complexity <strong>too well</strong>.
The programmer has surely an ease of programming, but design consideration should consider overloads generated by the network communication.</p>
<h4 id="implementation-of-the-rpc">Implementation of the RPC<a class="anchor" aria-hidden="true" hidden href="#implementation-of-the-rpc">#</a></h4>
<ul>
<li>How to determine location and identity of the Callee?</li>
<li>How to implement semantics for address argument, if there is no shared addressing space?</li>
<li>How to handle failures of network communication?</li>
</ul>
<p>These problems are solved with the following key features:</p>
<ul>
<li><strong>Stubs</strong>: basically generated interfaces that describe the kind of data that is transferred or received. This enables linkage between the two procedures.
<ul>
<li>We need a <strong>IDL (interface definition language)</strong>: to define the language-agnostic interface.</li>
</ul>
</li>
<li><strong>Serialization/Marshaling</strong>: the data is transmitted in a format good for transmission, but we would need to unpack and move it again to a processable format.</li>
<li><strong>Binding</strong>: knowing IP address and port, having some discovery service that tells you where the remote server is, this is called the <em>name and directory service</em> (big problem).</li>
</ul>
<p>The implementation still uses a format introduced in the 90'
</p><figure class="center">
<img src="/images/notes/Communication APIs-20250408114101664.webp" style="width: 100%" alt="Communication APIs-20250408114101664" title="Communication APIs-20250408114101664">
<figcaption><p style="text-align:center;">Distributed Computing Environment, a RPC framework from the 90'</p></figcaption>
</figure><p></p>
<h4 id="rpc-is-client-server">RPC is Client-Server<a class="anchor" aria-hidden="true" hidden href="#rpc-is-client-server">#</a></h4>
<p>RPC is quite a low level construct to build these systems.  Easy to use, hiding complexity from the programmer, yet it was build with the classical client-server architecture in mind, which sometimes might be outdated in modern communication frameworks.
Nice thing about the RPC structure is that it is <strong>fast</strong>.
Serialization is often costly, costs about 6% of the cycles of a server (see <a href="https://dl.acm.org/doi/10.1145/2749469.2750392">(Kanev et al. 2015)</a>).</p>
<h3 id="design-considerations">Design Considerations<a class="anchor" aria-hidden="true" hidden href="#design-considerations">#</a></h3>
<h4 id="transparency-of-server-and-call">Transparency of server and call<a class="anchor" aria-hidden="true" hidden href="#transparency-of-server-and-call">#</a></h4>
<p>This section is from a design perspective: would a developer be better off if calling local procedures and remote procedures uses the exact same code?</p>
<ul>
<li>YES -> meaning it is easier to implement, easier to manage the code
<ul>
<li>Today we know that hiding this is a <strong>bad idea</strong>. I have also personal experience for this thing. Ask me about it offline.</li>
</ul>
</li>
<li><strong>NO</strong> -> meaning that the developer <em>should be aware</em> of the fact that it is a remote procedure call, and should be aware of the network communication and the possible errors that can happen, which could directly affect the performance and structure of the code.</li>
</ul>
<p>Should the position of the remote server be transparent?</p>
<ul>
<li>YES -> the system should be able to move the server around easily, and the client should not care about it, good for fault tolerance too.</li>
<li><strong>NO</strong> -> the client should be aware of the fact that the server is remote. This should make the design easier, with less overhead of the configs.</li>
</ul>
<h4 id="failure-semantics-in-rpc">Failure Semantics in RPC<a class="anchor" aria-hidden="true" hidden href="#failure-semantics-in-rpc">#</a></h4>
<ul>
<li><strong>High latency</strong>: you need to wait for the other to answer</li>
<li><strong>Tight coupling</strong>: the two systems need each other to function.</li>
</ul>
<p>We have partly analyzed this concept in <a href="/notes/message-passing/">Message Passing</a>.</p>
<p>You should also define some <strong>failure semantics</strong>, this is usually implemented by attaching ids to the messages.</p>
<ul>
<li><em>At least once</em>: the client makes the call again until it succeeds and receives a response. This is usually implemented in the client side.</li>
<li><em>At most once</em>: the server remembers the calls, and executes the call only once, even if it is called multiple times. This is usually implemented on the server side.</li>
<li><em>Exactly once</em>: Mix of the previous two: it needs both the client and the server.</li>
</ul>

<h4 id="grpc">gRPC<a class="anchor" aria-hidden="true" hidden href="#grpc">#</a></h4>
<p>Now gRPC is built on top of HTTP most of the times, easier to handle discovery and reliability of the communication.</p>
<ul>
<li>Often used for <strong>performance critical systems</strong>, in the cloud.</li>
<li>More <strong>scalable</strong> compared to classical RPC</li>
<li>More <strong>efficient</strong> compared to REST.
Over time, implementations of RPC have evolved to adopt ideas from REST (see <a href="#rest">#REST</a>) such as gRPC (open source, initially from Google)</li>
<li>gRPC uses a standardized representation for serialization (Protocol Buffers, also from Google). Several such representations have been created by cloud providers, e.g:
<ul>
<li>Ion (Amazon)</li>
<li>Apache Thrift (Facebook/Meta)</li>
<li>Microsoft Bond</li>
<li>Operates on top of HTTP/2 instead of TCP/IP</li>
</ul>
</li>
</ul>
<h2 id="rest">REST<a class="anchor" aria-hidden="true" hidden href="#rest">#</a></h2>
<p>See <a href="/notes/http-e-rest/">HTTP e REST</a>.
The thing to remember is that</p>
<ul>
<li>Rest is more scalable and interoperable compared to RPC.</li>
<li>Might suffer from long <strong>chain of calls</strong>, good design patterns help in this context.</li>
</ul>
<h2 id="design-patterns-for-the-cloud">Design Patterns for the Cloud<a class="anchor" aria-hidden="true" hidden href="#design-patterns-for-the-cloud">#</a></h2>
<p>This is a different form of design patterns compared to <a href="/notes/design-patterns/">Design patterns</a> for software engineering, nonetheless important.</p>
<blockquote>
<p>REST becomes problematic when used in large-scale multi-tier settings as interaction becomes cumbersome and requires too many steps in environments where network latency matters (REST was based on the web and HTTP, not for a data center)</p>
</blockquote>
<p>A chain of too many calls would make the service quite slow.</p>
<h4 id="materialized-view">Materialized view<a class="anchor" aria-hidden="true" hidden href="#materialized-view">#</a></h4>
<p>Since most services just retrieve data, it might be helpful to put a <strong>cache in the middle</strong> to speed up the communication (so that I don't need to send a request).
Often this is done in the form of a materialized view of the needed information.
This is a classical design pattern, we also talked about it briefly when we studied <a href="/notes/relational-algebra/">Relational Algebra</a> and also in <a href="/notes/apache-spark/">Apache Spark</a> when caching intermediate results.</p>
<h4 id="service-aggregator">Service Aggregator<a class="anchor" aria-hidden="true" hidden href="#service-aggregator">#</a></h4>
<p>The idea is that instead of a complex sequence of calls, implement a single <strong>centralized</strong> service whose role is just to make calls. So that there are no complex dependencies anymore, only one with the service aggregator.</p>
<p><img src="/images/notes/Communication APIs-20250412181623539.webp" style="width: 100%" class="center" alt="Communication APIs-20250412181623539"></p>
<h4 id="message-queues">Message Queues<a class="anchor" aria-hidden="true" hidden href="#message-queues">#</a></h4>
<p>We can <strong>decouple</strong> the services using queues.
When the provided requests does not match the consuming rate, this is a good pattern.</p>
<p>The drawback is that we are adding some <strong>latency</strong>, the benefit is that we are <strong>decoupling</strong> the services, which brings an easier development.
It also allows for a transactional manner of storing requests, so that these can be <strong>persisted</strong> and audited.</p>
<p><img src="/images/notes/Communication APIs-20250412181811002.webp" style="width: 100%" class="center" alt="Communication APIs-20250412181811002"></p>
<h2 id="message-queuing-services">Message Queuing Services<a class="anchor" aria-hidden="true" hidden href="#message-queuing-services">#</a></h2>
<h3 id="basic-properties">Basic Properties<a class="anchor" aria-hidden="true" hidden href="#basic-properties">#</a></h3>
<p>Ordering or messages and charging can be defined by the user.
Also dropping queues etc.</p>
<h4 id="basic-functions-of-queues">Basic Functions of Queues<a class="anchor" aria-hidden="true" hidden href="#basic-functions-of-queues">#</a></h4>
<p>A message queuing system typically</p>
<ul>
<li>Provides queues for clients to PUT and GET messages</li>
<li>Stores the messages persistently to guarantee they survive failures</li>
<li>Makes the insertion and deletion of messages <strong>transactional</strong></li>
<li>Monitors queues and accesses for access control, scalability, routing, etc.</li>
<li>Manages the creation, deletion, and access to messages and queues</li>
</ul>
<h4 id="persistence-of-messages">Persistence of messages<a class="anchor" aria-hidden="true" hidden href="#persistence-of-messages">#</a></h4>
<p>A good thing is that now queues can be persisted in the queuing system. Receivers can read when service is up.
Services can <strong>multiplex</strong> requests to this system. This is a great thing with regards to the disassociated scaling of the resources.</p>
<h3 id="implementation-of-queuing-systems">Implementation of Queuing systems<a class="anchor" aria-hidden="true" hidden href="#implementation-of-queuing-systems">#</a></h3>
<h4 id="database-implementation">Database implementation<a class="anchor" aria-hidden="true" hidden href="#database-implementation">#</a></h4>
<p>We can implement it using a database, so messages are simply entries in this database.
We have transactions on this database. The important thing to remember is that messages are <strong>persistent</strong> and consumed in this manner. Separate domain with respect to producer and consumer of these messages.</p>
<h4 id="azure-store-queues">Azure Store Queues<a class="anchor" aria-hidden="true" hidden href="#azure-store-queues">#</a></h4>
<p>It is just building these queues on top of cloud buckets (see <a href="/notes/cloud-storage/">Cloud Storage</a>).
This service provides</p>
<ul>
<li><strong>No ordering</strong></li>
<li>Elimination after 7 days of the message</li>
<li>Only pay for <em>storage space.</em></li>
<li>Large capacity, and HTTP interfaces.</li>
<li>HTTP access.</li>
<li>Limited persistence.</li>
</ul>
<h4 id="amazon-sqs">Amazon SQS<a class="anchor" aria-hidden="true" hidden href="#amazon-sqs">#</a></h4>
<p>With this system we have two possible queues:</p>
<ul>
<li>Best effort ordering You have <em>at least once</em> semantics.</li>
<li>FIFO queues with <strong>extraction</strong> (different pricings), here is exactly once semantics.</li>
<li>Charged per million requests (standard queues are cheaper than fifo) and GB of readout data.</li>
<li><strong>Elastic deployment</strong>, with multiple copies across availability zones.</li>
<li>No volume limitation.</li>
</ul>
<h4 id="azure-service-bus-queues">Azure Service Bus Queues<a class="anchor" aria-hidden="true" hidden href="#azure-service-bus-queues">#</a></h4>
<p>Based on a message broker, it provides the typical features of a message queuing system:</p>
<ul>
<li>FIFO delivery</li>
<li>Transactional access</li>
<li>Duplicate detection</li>
<li>Supports queue partitioning across several message brokers (for scalability)</li>
<li>Session IDs used to <em>correlate messages</em> that belong together (correlation IDs)</li>
<li>Size of the queue is limited.</li>
<li>Charged on a base cost and per operation costs.</li>
</ul>
<h4 id="publish-and-subscribe">Publish and Subscribe<a class="anchor" aria-hidden="true" hidden href="#publish-and-subscribe">#</a></h4>
<p><img src="/images/notes/Communication APIs-20250412184341269.webp" style="width: 100%" class="center" alt="Communication APIs-20250412184341269"></p>
<h4 id="azure-event-grid">Azure Event Grid<a class="anchor" aria-hidden="true" hidden href="#azure-event-grid">#</a></h4>
<p>This is an example of a publish-subscribe based service, but more general: event based systems is an architectural approach to distributed computing, while publish and subscribe is an <em>implementation</em>.
Guarantees:</p>
<ul>
<li>Event delivery is in 24 hours with retries.</li>
<li>Notifications that can be sent to many destinations.</li>
<li>This is event based, so a little different compared to queues.</li>
</ul>
