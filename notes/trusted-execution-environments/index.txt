2f:T13041,<h2 id="trusted-execution-environments-and-the-secure-computation-trust-spectrum">Trusted Execution Environments and the Secure Computation Trust Spectrum<a class="anchor" aria-hidden="true" hidden href="#trusted-execution-environments-and-the-secure-computation-trust-spectrum">#</a></h2>
<h3 id="motivation-foundational-context">Motivation &#x26; Foundational Context<a class="anchor" aria-hidden="true" hidden href="#motivation-foundational-context">#</a></h3>
<p>How do you verify that a computation was performed correctly — and privately — when you don't control the machine? This question sits at the intersection of Cryptography, Hardware Security, Mechanism Design, and Multi-Agent Systems. Different answers trade off trust assumptions, performance, and expressiveness, forming a <strong>trust spectrum</strong> that has deep consequences for agent coordination, blockchain protocols, and verifiable AI.</p>
<p>The key insight: trust in computation is not binary. It ranges from "trust only mathematics" (ZK proofs) to "trust the hardware manufacturer" (TEEs) to "trust a set of humans" (multisigs). Each point on this spectrum corresponds to a different failure mode, a different cost profile, and a different set of mechanism design constraints.</p>
<h3 id="the-secure-computation-trust-spectrum">The Secure Computation Trust Spectrum<a class="anchor" aria-hidden="true" hidden href="#the-secure-computation-trust-spectrum">#</a></h3>
<h4 id="overview">Overview<a class="anchor" aria-hidden="true" hidden href="#overview">#</a></h4>
<p>Every system that claims to perform private or verifiable computation must answer: <em>who or what do you trust, and what happens when that trust is violated?</em></p>

<h4 id="the-trustperformance-tradeoff">The Trust–Performance Tradeoff<a class="anchor" aria-hidden="true" hidden href="#the-trustperformance-tradeoff">#</a></h4>
<p>This is the fundamental tension in secure computation:</p>
<p><span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6944em;"></span><span class="mord text"><span class="mord">Trustlessness</span></span><span class="mspace" style="margin-right:0.2778em;"></span><span class="mrel">∝</span><span class="mspace" style="margin-right:0.2778em;"></span></span><span class="base"><span class="strut" style="height:1.1901em;vertical-align:-0.345em;"></span><span class="mord"><span class="mopen nulldelimiter"></span><span class="mfrac"><span class="vlist-t vlist-t2"><span class="vlist-r"><span class="vlist" style="height:0.8451em;"><span style="top:-2.655em;"><span class="pstrut" style="height:3em;"></span><span class="sizing reset-size6 size3 mtight"><span class="mord mtight"><span class="mord text mtight"><span class="mord mtight">Performance</span></span></span></span></span><span style="top:-3.23em;"><span class="pstrut" style="height:3em;"></span><span class="frac-line" style="border-bottom-width:0.04em;"></span></span><span style="top:-3.394em;"><span class="pstrut" style="height:3em;"></span><span class="sizing reset-size6 size3 mtight"><span class="mord mtight"><span class="mord mtight">1</span></span></span></span></span><span class="vlist-s">​</span></span><span class="vlist-r"><span class="vlist" style="height:0.345em;"><span></span></span></span></span></span><span class="mclose nulldelimiter"></span></span></span></span></span></p>
<p>More precisely: the less you trust external entities (hardware, people, committees), the more computational work is required to achieve the same guarantees. ZK proofs trust nothing but math, but pay for it with orders-of-magnitude slowdown in proof generation. TEEs trust a chip manufacturer, but run at near-native speed.</p>
<p>This tradeoff is not merely engineering — it has deep implications for Mechanism Design:</p>
<ul>
<li>Systems with weaker trust assumptions (ZK, FHE) need fewer incentive mechanisms because the guarantees are mathematical, not behavioural.</li>
<li>Systems with stronger trust assumptions (TEE, multisig) require carefully designed incentive structures (staking, slashing, reputation) to make defection unprofitable. See Incentive Compatibility, Implementation Theory.</li>
</ul>

<h3 id="trusted-execution-environments-tees">Trusted Execution Environments (TEEs)<a class="anchor" aria-hidden="true" hidden href="#trusted-execution-environments-tees">#</a></h3>
<h4 id="core-concept">Core Concept<a class="anchor" aria-hidden="true" hidden href="#core-concept">#</a></h4>
<p>Intel SGX, ARM TrustZone, AMD SEV</p>
<p>A TEE is an isolated area within a device's processor that keeps code and data tamper-proof and confidential during execution. The isolation is enforced by hardware — the CPU itself encrypts the enclave's memory and refuses to let any external process (including the operating system, hypervisor, or someone with root access) read or modify it.</p>
<blockquote>
<p><strong>Definition</strong>: A TEE provides an isolated execution environment with three properties:</p>
<ol>
<li><strong>Confidentiality</strong>: Data inside the enclave is encrypted in memory; inaccessible to the host OS.</li>
<li><strong>Integrity</strong>: Code and data cannot be tampered with by external processes.</li>
<li><strong>Attestation</strong>: The enclave can produce a cryptographic proof that it is genuine hardware running a specific, untampered program.</li>
</ol>
</blockquote>
<p>The term "enclave" (used by Intel SGX) and "secure world" (used by ARM TrustZone) refer to the same concept: a hardware-isolated execution context.</p>

<h5 id="hardware-isolation-mechanisms">Hardware Isolation Mechanisms<a class="anchor" aria-hidden="true" hidden href="#hardware-isolation-mechanisms">#</a></h5>

<h5 id="the-attestation-protocol">The Attestation Protocol<a class="anchor" aria-hidden="true" hidden href="#the-attestation-protocol">#</a></h5>
<p>Attestation is the mechanism by which a TEE proves its identity and integrity to a remote verifier:</p>
<ol>
<li><strong>Measurement</strong>: At enclave launch, the CPU measures the Trusted Computing Base (boot firmware + OS kernel + application binaries) and stores the hash in secure hardware registers.</li>
<li><strong>Signing</strong>: The CPU signs this measurement using a private <strong>attestation key</strong> embedded in the silicon during manufacturing.</li>
<li><strong>Verification</strong>: A remote verifier checks the signature against the manufacturer's public key database, confirming: (a) the enclave runs on genuine hardware, and (b) the loaded code matches a known hash.</li>
</ol>

<h5 id="execution-flow-for-private-smart-contracts">Execution Flow for Private Smart Contracts<a class="anchor" aria-hidden="true" hidden href="#execution-flow-for-private-smart-contracts">#</a></h5>
<p>In the blockchain context (e.g., Secret Network, Oasis Network):</p>
<ol>
<li>Smart contract code is loaded into the enclave.</li>
<li>Enclave generates a keypair <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:1em;vertical-align:-0.25em;"></span><span class="mopen">(</span><span class="mord mathnormal">p</span><span class="mord mathnormal" style="margin-right:0.0315em;">k</span><span class="mpunct">,</span><span class="mspace" style="margin-right:0.1667em;"></span><span class="mord mathnormal">s</span><span class="mord mathnormal" style="margin-right:0.0315em;">k</span><span class="mclose">)</span></span></span></span> and publishes <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.8889em;vertical-align:-0.1944em;"></span><span class="mord mathnormal">p</span><span class="mord mathnormal" style="margin-right:0.0315em;">k</span></span></span></span> on-chain.</li>
<li>The CPU produces an attestation report, verifiable by anyone.</li>
<li>Users encrypt transaction inputs with <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.8889em;vertical-align:-0.1944em;"></span><span class="mord mathnormal">p</span><span class="mord mathnormal" style="margin-right:0.0315em;">k</span></span></span></span> before submitting.</li>
<li>Inside the enclave: decrypt inputs → execute contract → encrypt state updates.</li>
<li>Encrypted state posted on-chain. Only the enclave (and authorised viewers) can decrypt.</li>
</ol>
<p>The node operator sees only ciphertext at every stage — confidentiality holds even against the infrastructure provider.</p>
<h4 id="attack-surface-and-known-vulnerabilities">Attack Surface and Known Vulnerabilities<a class="anchor" aria-hidden="true" hidden href="#attack-surface-and-known-vulnerabilities">#</a></h4>
<p>TEEs have been broken multiple times. The trust in hardware is <strong>conditional</strong>, not absolute.</p>
<h5 id="side-channel-attacks">Side-Channel Attacks<a class="anchor" aria-hidden="true" hidden href="#side-channel-attacks">#</a></h5>
<p>The fundamental problem: TEEs share physical CPU resources (power rails, caches, execution units, branch predictors) with untrusted code. Information leaks through these shared channels.</p>















































<table><thead><tr><th align="left">Attack</th><th align="left">Year</th><th align="left">Mechanism</th><th align="left">Impact</th></tr></thead><tbody><tr><td align="left"><strong>Spectre / Meltdown</strong></td><td align="left">2018</td><td align="left">Speculative execution leaks data across isolation boundaries</td><td align="left">Data extraction from enclaves</td></tr><tr><td align="left"><strong>Foreshadow (L1TF)</strong></td><td align="left">2018</td><td align="left">L1 cache timing attack specific to SGX</td><td align="left">Full enclave secret extraction</td></tr><tr><td align="left"><strong>Plundervolt</strong></td><td align="left">2019</td><td align="left">CPU voltage manipulation induces faults inside SGX, bypassing integrity checks</td><td align="left">Key extraction from encrypted memory</td></tr><tr><td align="left"><strong>SGAxe / CrossTalk</strong></td><td align="left">2020</td><td align="left">Extracted SGX attestation keys via shared CPU buffers</td><td align="left">Undermines entire attestation model</td></tr><tr><td align="left"><strong>ÆPIC Leak</strong></td><td align="left">2022</td><td align="left">Architectural bug in APIC leaks enclave data</td><td align="left">Unpatched data extraction</td></tr><tr><td align="left"><strong>WireTap</strong></td><td align="left">2025</td><td align="left">DRAM bus interposition (physical)</td><td align="left">No software fix possible</td></tr></tbody></table>
<blockquote>
<p><strong>Critical observation</strong>: side-channel attacks exploit the fact that a TEE is a <em>partition</em> of a general-purpose processor, not a separate chip. A Secure Element (SE) — like the chip on a smart card or Ledger hardware wallet — is physically separate and does not share resources, making side-channel attacks much harder. But SEs are too limited to run general programs.</p>
</blockquote>
<h5 id="manufacturer-trust">Manufacturer Trust<a class="anchor" aria-hidden="true" hidden href="#manufacturer-trust">#</a></h5>
<p>The attestation key is embedded by the manufacturer. This creates irreducible trust dependencies:</p>
<ul>
<li><strong>Backdoor risk</strong>: Intel, AMD, ARM could be compelled by governments to introduce backdoors (see Intel Management Engine vulnerabilities).</li>
<li><strong>Supply chain attacks</strong>: Compromised manufacturing could embed rogue keys.</li>
<li><strong>Firmware vulnerabilities</strong>: Updates to microcode or firmware can introduce new attack vectors — and users must trust the manufacturer's update pipeline.</li>
</ul>
<blockquote>
<p>This is a fundamental difference from ZK proofs: a ZK proof is correct regardless of what hardware produced it. A TEE attestation is only trustworthy if the entire hardware supply chain is uncompromised.</p>
</blockquote>
<h3 id="fully-homomorphic-encryption-fhe">Fully Homomorphic Encryption (FHE)<a class="anchor" aria-hidden="true" hidden href="#fully-homomorphic-encryption-fhe">#</a></h3>
<h4 id="core-concept-1">Core Concept<a class="anchor" aria-hidden="true" hidden href="#core-concept-1">#</a></h4>
<p>Lattice-Based Cryptography, TFHE, BGV Scheme</p>
<p>FHE allows computation on encrypted data without decryption. Given ciphertexts <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:1em;vertical-align:-0.25em;"></span><span class="mord text"><span class="mopen">(</span><span class="mord mathnormal">b</span><span class="mclose">)</span></span></span></span>, one can compute <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:1em;vertical-align:-0.25em;"></span><span class="mord text"><span class="mord">Enc</span></span><span class="mopen">(</span><span class="mord <h4 id="performance-reality">Performance Reality<a class="anchor" aria-hidden="true" hidden href="#performance-reality">#</a></h4>
<p>FHE is often called the "holy grail" of encryption, but the performance overhead is severe:</p>
<blockquote>
<p>FHE is approximately <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.8141em;"></span><span class="mord">1</span><span class="mord"><span class="mord">0</span><span class="msupsub"><span class="vlist-t"><span class="vlist-r"><span class="vlist" style="height:0.8141em;"><span style="top:-3.063em;margin-right:0.05em;"><span class="pstrut" style="height:2.7em;"></span><span class="sizing reset-size6 size3 mtight"><span class="mord mtight">4</span></span></span></span></span></span></span></span></span></span></span> to <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.8141em;"></span><span class="mord">1</span><span class="mord"><span class="mord">0</span><span class="msupsub"><span class="vlist-t"><span class="vlist-r"><span class="vlist" style="height:0.8141em;"><span style="top:-3.063em;margin-right:0.05em;"><span class="pstrut" style="height:2.7em;"></span><span class="sizing reset-size6 size3 mtight"><span class="mord mtight">5</span></span></span></span></span></span></span></span></span></span></span> times slower than computing on unencrypted data (Feldmann et al., 2021; Microsoft SEAL benchmarks).</p>
</blockquote>
<p>This makes FHE impractical for most real-time applications. Current research focuses on:</p>
<ul>
<li><strong>Hardware acceleration</strong>: Dedicated FHE accelerators (DARPA DPRIVE program, Intel HEXL).</li>
<li><strong>Levelled FHE</strong>: Evaluate circuits of bounded multiplicative depth without bootstrapping.</li>
<li><strong>Noise management</strong>: Bootstrapping refreshes noise but is extremely expensive (<span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.5782em;vertical-align:-0.0391em;"></span><span class="mrel">></span><span class="mspace" style="margin-right:0.2778em;"></span></span><span class="base"><span class="strut" style="height:0.6444em;"></span><span class="mord">10</span></span></span></span>ms per gate).</li>
</ul>
<h4 id="fhe-in-blockchain">FHE in Blockchain<a class="anchor" aria-hidden="true" hidden href="#fhe-in-blockchain">#</a></h4>
<p>fhEVM, Zama, Fhenix</p>
<p><strong>Zama</strong> developed the fhEVM — a framework for encrypted smart contracts where contract state is FHE-encrypted. Operations (addition, comparison, transfer) happen homomorphically. The decryption key is split via threshold MPC across multiple parties.</p>
<p>The key management problem is fundamental: <em>someone</em> must hold the decryption key. FHE alone doesn't solve this — it pushes the trust to the key holder. This is why practical FHE systems combine FHE with MPC for distributed key management.</p>
<h4 id="post-quantum-security">Post-Quantum Security<a class="anchor" aria-hidden="true" hidden href="#post-quantum-security">#</a></h4>
<p>FHE schemes are based on Lattice-Based Cryptography (LWE, RLWE), which is believed to be resistant to quantum attacks (Shor's algorithm does not help). This is a significant advantage over pairing-based SNARKs and TEEs whose cryptographic components rely on classical assumptions.</p>

<h4 id="mpc-in-blockchain">MPC in Blockchain<a class="anchor" aria-hidden="true" hidden href="#mpc-in-blockchain">#</a></h4>
<p>Threshold Signatures, MPC Wallets, Nillion</p>
<p>Primary blockchain applications:</p>
<ul>
<li><strong>Threshold signatures</strong>: Split a signing key across <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.4306em;"></span><span class="mord mathnormal">n</span></span></span></span> parties; <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6151em;"></span><span class="mord mathnormal">t</span></span></span></span>-of-<span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.4306em;"></span><span class="mord mathnormal">n</span></span></span></span> must collaborate to sign. No single party ever holds the full key. Used by institutional custodians (Fireblocks, Coinbase).</li>
<li><strong>MPC-based key management</strong>: Distribute wallet keys without a single point of failure. More flexible than multisig (no on-chain footprint).</li>
<li><strong>MPC + FHE hybrid</strong>: Nillion orchestrates MPC, FHE, and ZK proofs depending on computation requirements. Zama's fhEVM uses threshold MPC for FHE decryption key management.</li>
</ul>
<h3 id="comparative-analysis">Comparative Analysis<a class="anchor" aria-hidden="true" hidden href="#comparative-analysis">#</a></h3>
<h4 id="trust-basis-math-vs-hardware-vs-people">Trust Basis: Math vs. Hardware vs. People<a class="anchor" aria-hidden="true" hidden href="#trust-basis-math-vs-hardware-vs-people">#</a></h4>
<p>The fundamental taxonomy of verifiable computation:</p>
<pre><code>              Trust basis
              
  Math-only ──────────────── Hardware ──── Social
      │                         │            │
   ZK proofs                  TEEs        Multisig
   FHE                    (attestation)   (M-of-N)
   MPC (threshold)
      │                         │            │
  Failure mode:            Failure mode:   Failure mode:
  Implementation bug       Side-channel,   Bribery,
  (≈ rare, fixable)        backdoor        coercion
                           (≈ hardware      (≈ social
                            lifecycle)       engineering)
</code></pre>
<blockquote>
<p><strong>For mechanism designers</strong>: the choice of trust basis determines the <strong>residual trust</strong> that must be covered by incentive mechanisms. A system built on ZK proofs needs minimal incentive design for correctness (the math handles it), but may need incentives for <em>liveness</em> (someone must generate the proofs). A system built on TEEs needs incentive mechanisms for both correctness (TEEs can lie if compromised) and liveness — plus an additional social layer of trust in the hardware supply chain. See Implementation Theory, Hurwicz Framework.</p>
</blockquote>

<h3 id="connection-to-mechanism-design-and-agent-coordination">Connection to Mechanism Design and Agent Coordination<a class="anchor" aria-hidden="true" hidden href="#connection-to-mechanism-design-and-agent-coordination">#</a></h3>
<h4 id="computational-trust-as-a-mechanism-design-problem">Computational Trust as a Mechanism Design Problem<a class="anchor" aria-hidden="true" hidden href="#computational-trust-as-a-mechanism-design-problem">#</a></h4>
<p>Mechanism Design, Implementation Theory, Hurwicz Framework</p>
<p>The trust spectrum maps directly onto Mechanism Design concepts. Consider a multi-agent system where agents must jointly compute a function (e.g., aggregate preferences, execute a trade, verify a claim):</p>
<blockquote>
<p><strong>The mechanism design question</strong>: given that agents may be self-interested (or adversarial), how do you design the computation infrastructure so that the outcome is correct, private, and incentive-compatible?</p>
</blockquote>
<p>The answer depends on what you trust:</p>






























<table><thead><tr><th align="left">Trust Basis</th><th align="left">IC Requirement</th><th align="left">Mechanism Implications</th></tr></thead><tbody><tr><td align="left"><strong>ZK proofs</strong></td><td align="left">Minimal — math guarantees correctness</td><td align="left">Incentives needed only for liveness (who runs the prover?)</td></tr><tr><td align="left"><strong>MPC</strong></td><td align="left">Moderate — threshold honesty assumed</td><td align="left">Requires either honest majority or economic incentives (staking, slashing) to keep collusion below threshold</td></tr><tr><td align="left"><strong>TEE</strong></td><td align="left">High — hardware integrity assumed</td><td align="left">Requires reputation systems, hardware audits, attestation verification infrastructure</td></tr><tr><td align="left"><strong>Multisig</strong></td><td align="left">Maximal — social trust required</td><td align="left">Requires governance, legal frameworks, social accountability</td></tr></tbody></table>
<blockquote>
<p>In the language of Hurwicz Framework: ZK proofs achieve a form of "strategy-proofness" for the verification step — the verifier has no profitable deviation from honest checking, because the proof is self-certifying. TEE-based systems require additional incentive layers to approximate this property.</p>
</blockquote>
<h4 id="agent-trust-and-coordination">Agent Trust and Coordination<a class="anchor" aria-hidden="true" hidden href="#agent-trust-and-coordination">#</a></h4>
<p>Multi-Agent Systems, Game Theory, Cooperation Theory, Team Theory</p>
<p>In multi-agent systems, the secure computation trust spectrum maps onto the question: <em>how much can agents trust each other's reported computations?</em></p>
<p>Consider a setting with <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.4306em;"></span><span class="mord mathnormal">n</span></span></span></span> autonomous AI agents that must coordinate (e.g., distributed DeFi protocol, multi-agent supply chain optimisation, federated learning):</p>
<p><strong>Scenario 1 — ZK-verified agents</strong>: Each agent produces a ZK proof alongside its output, attesting to correct computation. Other agents verify the proof before acting on the output. Trust is minimal — agents need not trust each other, only the proof system. This is analogous to the <strong>verifiable mechanism</strong> concept in Implementation Theory: the mechanism is self-enforcing because deviation is detectable.</p>
<p><strong>Scenario 2 — TEE-attested agents</strong>: Each agent runs inside a TEE and provides an attestation report. Other agents trust the attestation if they trust the hardware manufacturer. This is analogous to <strong>reputation-based trust</strong> in Repeated Games: the hardware manufacturer's reputation is the enforcement mechanism. But unlike reputation in iterated games, hardware vulnerabilities are discovered discretely and affect all instances simultaneously (correlated failure).</p>
<p><strong>Scenario 3 — MPC-coordinated agents</strong>: Agents engage in an MPC protocol, each contributing private inputs. Trust is distributed across the threshold. This maps to <strong>team theory</strong> (Marschak-Radner) with private information: agents jointly compute a function under incentive constraints, and the MPC protocol is the communication structure.</p>
<blockquote>
<p><strong>The correlated failure problem</strong>: TEEs introduce a unique risk absent in cryptographic approaches — all enclaves from the same manufacturer share the same vulnerability. A single side-channel discovery (e.g., Spectre) compromises <em>every</em> enclave simultaneously. In game-theoretic terms, this is a <strong>common shock</strong> — an exogenous event that correlates agent failures. Mechanism designers must account for this correlated structure when designing fallback systems. See Correlated Equilibrium, Common Knowledge.</p>
</blockquote>
<h4 id="verification-layers-for-autonomous-agents">Verification Layers for Autonomous Agents<a class="anchor" aria-hidden="true" hidden href="#verification-layers-for-autonomous-agents">#</a></h4>
<p>Verifiable AI, AI Safety, Multi-Agent Safety</p>
<p>For autonomous AI agents operating on-chain, the trust spectrum becomes a <strong>verification architecture</strong> question:</p>



































<table><thead><tr><th align="left">Layer</th><th align="left">Technology</th><th align="left">What It Verifies</th><th align="left">Latency</th></tr></thead><tbody><tr><td align="left"><strong>Real-time</strong></td><td align="left">TEE attestation</td><td align="left">"Agent is running claimed model on genuine hardware"</td><td align="left">Milliseconds</td></tr><tr><td align="left"><strong>Periodic</strong></td><td align="left">ZK proof of execution</td><td align="left">"Agent's outputs are consistent with claimed model and inputs"</td><td align="left">Minutes–hours</td></tr><tr><td align="left"><strong>Post-hoc</strong></td><td align="left">Fraud proof (optimistic)</td><td align="left">"No validator has contested the agent's outputs within the challenge window"</td><td align="left">Days</td></tr><tr><td align="left"><strong>Audit</strong></td><td align="left">Full re-execution + MPC</td><td align="left">"Independent parties jointly verified the computation with private inputs"</td><td align="left">Offline</td></tr></tbody></table>
<blockquote>
<p><strong>Design principle</strong>: layer faster-but-weaker guarantees (TEE) with slower-but-stronger guarantees (ZK), creating a <strong>defense-in-depth</strong> for agent trust. This is the mechanism design analogue of combining screening and monitoring in Principal-Agent Problems — the principal uses cheap, noisy monitoring (TEE attestation) for real-time oversight and expensive, precise auditing (ZK proofs) for periodic verification.</p>
</blockquote>
<h4 id="trust-spectrum-and-cooperation-theory">Trust Spectrum and Cooperation Theory<a class="anchor" aria-hidden="true" hidden href="#trust-spectrum-and-cooperation-theory">#</a></h4>
<p>Cooperation Theory, Evolutionary Game Theory, Signaling Theory</p>
<p>The trust technologies we've examined can be understood through the lens of Cooperation Theory — specifically, as different solutions to the <strong>cooperation problem</strong> in multi-agent systems:</p>
<blockquote>
<p>How do you sustain cooperation when agents have private information, conflicting incentives, and the ability to defect?</p>
</blockquote>
<p>Each technology corresponds to a different cooperation-enforcement mechanism:</p>






























<table><thead><tr><th align="left">Trust Technology</th><th align="left">Cooperation Mechanism</th><th align="left">Game-Theoretic Analogue</th></tr></thead><tbody><tr><td align="left"><strong>ZK Proofs</strong></td><td align="left">Verifiable commitment</td><td align="left">Costly signaling (Spence Signaling): the proof is a hard-to-forge signal of honest behavior. Producing a valid ZK proof is computationally expensive, creating a natural separation between honest and dishonest agents.</td></tr><tr><td align="left"><strong>TEE</strong></td><td align="left">Delegated enforcement</td><td align="left">Third-party arbitrator: Intel/AMD acts as an external enforcement mechanism, similar to a court in contract theory. The failure mode is that the "court" itself may be compromised.</td></tr><tr><td align="left"><strong>MPC</strong></td><td align="left">Distributed trust / mutual verification</td><td align="left">Conditional cooperation in Repeated Games: agents cooperate because they each hold a piece of the puzzle, and unilateral defection is detectable.</td></tr><tr><td align="left"><strong>Multisig</strong></td><td align="left">Social enforcement</td><td align="left">Ostrom's commons governance (Elinor Ostrom): cooperation sustained by social norms, reputation, and graduated sanctions among a known group.</td></tr></tbody></table>
<blockquote>
<p><strong>The deep connection</strong>: ZK proofs are the cryptographic analogue of <strong>credible commitment</strong> in game theory. Just as Schelling's commitment devices make defection physically impossible (burning bridges), a ZK proof makes false claims mathematically impossible. TEEs are a weaker commitment device — they make false claims <em>hardware-costly</em> rather than impossible. MPC distributes the commitment across agents, making defection require collusion. Each maps to a different equilibrium concept: ZK → dominant strategy; TEE → Nash equilibrium (with hardware assumptions); MPC → correlated equilibrium with threshold constraints.</p>
</blockquote>
<h3 id="blockchain-applications-of-tees">Blockchain Applications of TEEs<a class="anchor" aria-hidden="true" hidden href="#blockchain-applications-of-tees">#</a></h3>
<h4 id="private-smart-contracts">Private Smart Contracts<a class="anchor" aria-hidden="true" hidden href="#private-smart-contracts">#</a></h4>
<p>Secret Network, Oasis Network, Phala Network</p>
<p>Secret Network was the first blockchain to implement private smart contracts via TEEs. Using Intel SGX, it enables "Secret Contracts" where contract logic, inputs, outputs, and state are hidden from node operators. Only addresses are visible on-chain.</p>
<p><strong>Architecture</strong>: Each validator runs an SGX enclave. Contract state is encrypted with keys managed by a distributed Key Management Committee (KMC). If a KMC node is compromised, its access can be revoked through governance. Short-term keys are rotated frequently to limit breach impact.</p>
<p><strong>Tradeoff vs. ZK private contracts</strong> (e.g., Aztec Network): TEE-based privacy runs at near-native speed and supports arbitrary contract logic today. ZK-based privacy is slower and requires circuit compilation, but doesn't depend on hardware trust. Long-term, ZK is likely to prevail as zkEVMs mature.</p>
<h4 id="cross-chain-bridges">Cross-Chain Bridges<a class="anchor" aria-hidden="true" hidden href="#cross-chain-bridges">#</a></h4>
<p>Traditional bridges use multisig committees — a social trust model with known failure modes (Ronin: <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.8889em;vertical-align:-0.1944em;"></span><span class="mord">625</span><span class="mord mathnormal" style="margin-right:0.109em;">M</span><span class="mpunct">,</span><span class="mspace" style="margin-right:0.1667em;"></span><span class="mord mathnormal" style="margin-right:0.1389em;">W</span><span class="mord mathnormal" style="margin-right:0.0278em;">or</span><span class="mord mathnormal">mh</span><span class="mord mathnormal">o</span><span class="mord mathnormal" style="margin-right:0.0197em;">l</span><span class="mord mathnormal">e</span><span class="mspace" style="margin-right:0.2778em;"></span><span class="mrel">:</span></span></span></span>320M). TEE-secured bridges place signing keys and validation logic inside enclaves, reducing the trust assumption from "M-of-N humans are honest" to "the enclave hardware is uncompromised." This is a strict improvement on the trust spectrum, though not as strong as ZK-verified bridges (which are still largely in development).</p>
<h4 id="oracle-networks">Oracle Networks<a class="anchor" aria-hidden="true" hidden href="#oracle-networks">#</a></h4>
<p>Chainlink, Town Crier</p>
<p>Oracles deliver off-chain data to smart contracts. TEEs (specifically Town Crier, which influenced Chainlink's design) ensure that the data feed is fetched and delivered inside an enclave — the oracle operator cannot tamper with the data between source and chain. Attestation proves the data came from the claimed HTTPS endpoint.</p>
<h4 id="confidential-ai-inference">Confidential AI Inference<a class="anchor" aria-hidden="true" hidden href="#confidential-ai-inference">#</a></h4>
<p>NVIDIA Confidential Computing, Verifiable AI</p>
<p>NVIDIA's Hopper and Blackwell GPUs include confidential computing enclaves, enabling private ML inference: the model weights stay encrypted in GPU memory, inputs are encrypted in transit, and the inference runs inside the enclave. This is orders of magnitude faster than proving the same inference in a ZK circuit (which is currently infeasible for large models).</p>
<blockquote>
<p><strong>TEE + ZK hybrid for AI</strong>: Run inference in a GPU enclave for speed; generate a ZK proof that the TEE's output is consistent with the attested model for trustless verification. This is the emerging architecture for verifiable AI agents.</p>
</blockquote>
<h3 id="critical-pitfalls-warnings">Critical Pitfalls &#x26; Warnings<a class="anchor" aria-hidden="true" hidden href="#critical-pitfalls-warnings">#</a></h3>
<h4 id="tees-are-necessary-but-not-sufficient">TEEs Are Necessary But Not Sufficient<a class="anchor" aria-hidden="true" hidden href="#tees-are-necessary-but-not-sufficient">#</a></h4>
<blockquote>
<p>TEEs provide <em>conditional</em> security — they work if the hardware is uncompromised. ZK proofs provide <em>unconditional</em> computational security — they work regardless of the hardware.</p>
</blockquote>
<p>A system that relies solely on TEEs inherits every past and future vulnerability of the underlying silicon. This is an irreducible risk that cannot be mitigated by software — the WireTap attack (CCS 2025) demonstrated a physical DRAM bus interposition that no firmware patch can fix.</p>
<h4 id="attestation-trust">Attestation ≠ Trust<a class="anchor" aria-hidden="true" hidden href="#attestation-trust">#</a></h4>
<p>Attestation proves the enclave is running specific code on genuine hardware. It does <em>not</em> prove:</p>
<ul>
<li>The code itself is bug-free or correct.</li>
<li>The hardware has no undiscovered vulnerabilities.</li>
<li>The manufacturer has not been coerced into providing backdoor access.</li>
<li>The attestation key has not been extracted (as in SGAxe).</li>
</ul>
<blockquote>
<p><strong>Analogy to Signaling Theory</strong>: attestation is a <em>signal</em>, not a <em>proof</em>. Its credibility depends on the signaler's (manufacturer's) incentives and the cost of forgery. When the cost of forgery decreases (via discovered vulnerabilities), the signal's informativeness degrades — a standard result from Spence Signaling models.</p>
</blockquote>
<h4 id="the-correlated-failure-risk">The Correlated Failure Risk<a class="anchor" aria-hidden="true" hidden href="#the-correlated-failure-risk">#</a></h4>
<p>All Intel SGX enclaves share the same silicon design. A single vulnerability discovery affects every deployed enclave simultaneously. This is unlike MPC (where failure requires independent corruption of <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6151em;"></span><span class="mord mathnormal">t</span></span></span></span> parties) or ZK (where failure requires breaking a mathematical assumption).</p>
<p>In mechanism design terms: TEE failures are <strong>correlated shocks</strong>, not independent failures. Systems built on TEEs must have fallback mechanisms (ZK proof generation, optimistic challenge windows, MPC-based key recovery) that activate when the hardware trust assumption is violated.</p>
<h3 id="summary-choosing-the-right-trust-model">Summary: Choosing the Right Trust Model<a class="anchor" aria-hidden="true" hidden href="#summary-choosing-the-right-trust-model">#</a></h3>
<p>The choice of secure computation technology is ultimately a mechanism design decision:</p>
<ol>
<li><strong>What is the cost of failure?</strong> High-stakes systems (bridges, custody, AI agent verification) should layer multiple trust bases.</li>
<li><strong>What is the latency budget?</strong> Real-time systems (DeFi, gaming) may need TEE for speed, with ZK as a periodic backstop.</li>
<li><strong>What is the threat model?</strong> Nation-state adversaries can compromise hardware supply chains; pure-crypto approaches (ZK, FHE) are more robust here.</li>
<li><strong>What is the coordination structure?</strong> Multi-party settings naturally suit MPC; single-prover settings suit ZK or TEE.</li>
<li><strong>What is the quantum timeline?</strong> Long-lived systems should prefer hash-based (STARKs) or lattice-based (FHE) constructions.</li>
</ol>
<blockquote>
<p><strong>The emerging consensus</strong>: no single technology suffices. The future of secure computation is compositional — hybrid architectures that select the optimal trust–performance point for each sub-computation. The mechanism designer's job is to structure these compositions so that the overall system is incentive-compatible, resilient to correlated failures, and auditable across trust boundaries.</p>
</blockquote>
