2a:T1669,<p>CDNs are intermediary servers that replicate read intensive data to provide better performance when user requests them.
A close relative of CDNs is edge computing (e.g. gaming stations) where lots of computation is done directly close to the user.</p>
<h4 id="types-of-cdns">Types of CDNs<a class="anchor" aria-hidden="true" hidden href="#types-of-cdns">#</a></h4>
<p>Mainly three types of CDNs:</p>
<ul>
<li>Highly distributed ones. -> <strong>Akamai</strong></li>
<li>Database based ones.</li>
<li>Ad-hoc CDNs.
<img src="/images/notes/Content Delivery Networks-20250403143719218.webp" style="width: 100%" class="center" alt="Content Delivery Networks-20250403143719218"></li>
</ul>
<h4 id="advantages-and-disadvantages">Advantages and disadvantages<a class="anchor" aria-hidden="true" hidden href="#advantages-and-disadvantages">#</a></h4>
<p>The main reason we use CDNs is to lower the value of latency: we are in fact bringing the data closer to the user.
We have much less data in length to be transmitted.
Yet we have some disadvantages too:</p>
<ul>
<li>They usually are not able to provide much storage</li>
<li>Difficult to manage and coordinate.</li>
</ul>
<h3 id="cache-design">Cache Design<a class="anchor" aria-hidden="true" hidden href="#cache-design">#</a></h3>
<p>If you want to optimize for computer caches, see <a href="/notes/cache-optimization/">Cache Optimization</a>.</p>
<h4 id="the-reuse-distance">The Reuse Distance<a class="anchor" aria-hidden="true" hidden href="#the-reuse-distance">#</a></h4>
<p>The reuse distance of an element in a certain access sequence is the number of middle elements until we get that element again. In classical LRU caches, with this value you can compute the miss rate.</p>
<h4 id="increase-hit-ratio">Increase Hit Ratio<a class="anchor" aria-hidden="true" hidden href="#increase-hit-ratio">#</a></h4>
<p>If we increase the cache hit ratio, we will have indeed better performance. There are some standard ways that are known in the academy:</p>
<ul>
<li>Increase cache size</li>
<li>Design better algorithms
<img src="/images/notes/Content Delivery Networks-20250403144237728.webp" style="width: 100%" class="center" alt="Content Delivery Networks-20250403144237728"></li>
</ul>
<h4 id="lazy-promotion-and-quick-demotion">Lazy promotion and Quick Demotion<a class="anchor" aria-hidden="true" hidden href="#lazy-promotion-and-quick-demotion">#</a></h4>
<p>These are two design principles that have proven to work when designing efficient caches.
<strong>Lazy promotion</strong> retain popular objects with minimal effort</p>
<ul>
<li>Improve throughput due to less computation</li>
<li>Improve efficiency due to more information at eviction
<ul>
<li>Reduces the frequency of recency updates, potentially improving performance.</li>
</ul>
</li>
<li>The idea is to wait to upgrade an element to <em>VIP</em> member in the main queue, and wait to record a certain number of accesses, or access pattern that enable it to move up.</li>
</ul>
<p><strong>Quick Demotion</strong>: Remove unpopular objects fast, such as one-hit-wonders.</p>
<h4 id="sieve-caching">Sieve Caching<a class="anchor" aria-hidden="true" hidden href="#sieve-caching">#</a></h4>
<p>SIEVE considers both <strong>how frequently</strong> an item has been accessed (frequency) and <strong>how recently</strong> it was accessed (recency). However, it adds an interesting layer: it tries to <em>actively</em> identify and remove items that are likely to be accessed only once or a very limited number of times. These are often referred to as "one-hit wonders" or "transient" items.</p>
<p>The following represents the algorithm in short:
<img src="https://cachemon.github.io/SIEVE-website/assets/images/illustrations/sieve_diagram.gif" alt="Example of the algorithm"></p>
<h3 id="load-balancing">Load Balancing<a class="anchor" aria-hidden="true" hidden href="#load-balancing">#</a></h3>
<p>When we have lots of requests, we want to find an optimal way to split these among the available resources.
Another example of <strong>automatic load balancing</strong> are key value stores (see <a href="/notes/cloud-storage/#key-value-stores">Cloud Storage#Key-value stores</a>).</p>
<h4 id="global-load-balancing">Global Load Balancing<a class="anchor" aria-hidden="true" hidden href="#global-load-balancing">#</a></h4>
<ul>
<li>Avoid TCP set up delay (latency)</li>
<li>Use of DNS caching.
AKAMAI uses a kind of iterative DNS discovery (see <a href="/notes/livello-applicazione-e-socket/#domain-name-system">Livello applicazione e socket#Domain Name System</a>).
In this DNS type, it makes a request, and it tells you other DNS name system to contact to get the IP.
When you go the IP, you can make the HTTP request to the akamai server for the resource.
This server contacts the content provider if there is no copy available, else it does it correctly.</li>
</ul>
<h4 id="anycast">Anycast<a class="anchor" aria-hidden="true" hidden href="#anycast">#</a></h4>
<p>With Anycast, we have a group of servers that share the IP. When a user makes a request, routers are able to deliver it to the server closer <strong>geographically</strong>.</p>
<p>Advantages</p>
<ul>
<li>No extra round trips</li>
<li>Route to <strong>nearby server</strong>
Disadvantages</li>
<li>Does not consider network or server load (e.g. a server in region could have higher load compared to another server, with this method we don't have any load balancing based on load)</li>
<li>Different packets may go to different servers</li>
<li>Used only for simple request-response apps</li>
</ul>
