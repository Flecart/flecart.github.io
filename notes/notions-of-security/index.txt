1e:T3551,<h2 id="ciaa-principles-of-security">CIAA principles of security<a class="anchor" aria-hidden="true" hidden href="#ciaa-principles-of-security">#</a></h2>
<p>We have already  outlined these principles in <a href="/notes/sicurezza-delle-reti/">Sicurezza delle reti</a> and talked about the concepts of authentication and integrity. Here we try to deepen these concepts and delve a little bit more on the attack vectors. This note mainly focuses on the principles summarized by the acronyms CIA and AAA.</p>
<h3 id="confidentiality">Confidentiality<a class="anchor" aria-hidden="true" hidden href="#confidentiality">#</a></h3>
<p>This is one concerns about the secrecy of the sent message. We do not want others to be able to access and read what we are doing.</p>
<h4 id="eavesdropping">Eavesdropping<a class="anchor" aria-hidden="true" hidden href="#eavesdropping">#</a></h4>
<p>This is an example of attack of confidentiality. The setting is usually like this: Eve that intercepts the message sent by each other.
For example in network security, it is quite easy to eavesdrop with Wireshark or similars.</p>
<h3 id="integrity">Integrity<a class="anchor" aria-hidden="true" hidden href="#integrity">#</a></h3>
<p>Integrity concerns with message tampering. The received message should be the same as the sent one (man in the middle are common attacks).</p>
<h4 id="authentication">Authentication<a class="anchor" aria-hidden="true" hidden href="#authentication">#</a></h4>
<p>Authentication is important when we <strong>need to know</strong> to whom we are talking to. We should need to be sure that that is exactly the person (or the machine) we are trying to connect (or talk to). In this framework it is about integrity.
For more in depth analysis see <a href="/notes/user-authentication/">User authentication</a>.</p>
<h4 id="spoofing-attacks">Spoofing attacks<a class="anchor" aria-hidden="true" hidden href="#spoofing-attacks">#</a></h4>
<p>When an attacker authenticates as another user.</p>
<h4 id="manipulation-attacks">Manipulation attacks<a class="anchor" aria-hidden="true" hidden href="#manipulation-attacks">#</a></h4>
<p>This is tampering.</p>
<h3 id="availability">Availability<a class="anchor" aria-hidden="true" hidden href="#availability">#</a></h3>
<p>The system should be available, that is accessible by its users. Attacks that prevent this cornerstone are attacks that make system less performant or inaccessible to the user.</p>
<h4 id="denial-of-service-attacks">Denial of service attacks<a class="anchor" aria-hidden="true" hidden href="#denial-of-service-attacks">#</a></h4>
<p>For example if you have limited number of ports, a common example of denial of service attack is the <strong>Syn flooding</strong> where multiple services ask to open a TCP connection, but it doesn't continue with the communication, leaving the port occupied but useless.</p>
<h3 id="anonymity">Anonymity<a class="anchor" aria-hidden="true" hidden href="#anonymity">#</a></h3>
<p>On the internet we <strong>are not anonymous</strong> we are always tracked by ISP, cookies and many other strategies that I am not even aware of.
This is a problem we we want to be anonymous, so how can we reach this target??</p>
<h4 id="anonymity-by-proxy">Anonymity by proxy<a class="anchor" aria-hidden="true" hidden href="#anonymity-by-proxy">#</a></h4>
<p>We just use another computer to repeat my information, this computer doesn't have access to the underlying information, but it substitutes his IP to ours, so the end receiver doesn't exactly know where the initial message comes from.</p>

<p>See <a href="/notes/sicurezza-os/">Sicurezza OS</a>, we describe these notions more in detail, but in the context of computer operating systems.</p>

<h2 id="other-notions">Other Notions<a class="anchor" aria-hidden="true" hidden href="#other-notions">#</a></h2>
<h3 id="basic-notions">Basic Notions<a class="anchor" aria-hidden="true" hidden href="#basic-notions">#</a></h3>
<p>In this section, we introduce some of the most common definitions in security.</p>
<h4 id="trusted-computing-base">Trusted Computing Base<a class="anchor" aria-hidden="true" hidden href="#trusted-computing-base">#</a></h4>
<p>A trusted computing base (TCB) defines the parts of the system that are <strong>trusted</strong> by the host or guest (could be hardware or also software).
It is important to keep in mind whose perspective are we taking.</p>
<p>In cloud computing systems hypervisor hardware and host software are trusted by the host, but not guest software.</p>
<p><img src="/images/notes/Theoretical Notions of Security-20250520112631542.webp" style="width: 100%" class="center" alt="Theoretical Notions of Security-20250520112631542"></p>
<h4 id="attack-surface">Attack Surface<a class="anchor" aria-hidden="true" hidden href="#attack-surface">#</a></h4>
<p>The attack surface is the sum of the different points (attack vectors) in a computing device or network that are accessible to an unauthorized user (attacker).
In cloud VMs examples are:</p>
<ul>
<li>CPU instruction set for the VM</li>
<li>Hypercall API (virtual I/O, memory management, etc.)</li>
<li>Network interfaces</li>
</ul>
<h4 id="defense-in-depth">Defense in Depth<a class="anchor" aria-hidden="true" hidden href="#defense-in-depth">#</a></h4>
<p>This is the idea where you need to be able to exploit <strong>multiple layers</strong> in order to actually gain control of the machine. One example is hypervisor models, for example with firecracker or QEMU, you should need to exploit also the hypervisor to gain control on the machine.</p>
<h4 id="threat-model">Threat Model<a class="anchor" aria-hidden="true" hidden href="#threat-model">#</a></h4>
<p>A threat model is a structured representation of all the information that affects the security of an application. Usually it is more used in academic contexts to analyze security principles of the system.
It includes:</p>
<ul>
<li>the assets to be protected</li>
<li>the potential threats to those assets</li>
<li>What the attacker <strong>can and cannot</strong> do (e.g. the infinite computational power assumptions, we explored in <a href="/notes/asymmetric-cryptography/">Asymmetric Cryptography</a> proofs).</li>
<li>This helps also to define <em>out-of-scope</em> threats.</li>
</ul>
<p>There is a real example of 2018 attacks on side channels (see <a href="https://meltdownattack.com/">Spectre and Meltdown</a>. They proved that the Intel CPU micro-architecture is vulnerable to side-channel attacks. They exploited the speculative execution feature of modern processors to leak sensitive data (kernel memory) from the CPU cache, in user mode, there was <strong>no isolation anymore</strong>. This was a huge problem for cloud computing, as it affected the security of virtual machines running on shared hardware.
The whole industry had to pivot to care about side-channel attacks.</p>
<h4 id="memory-safety">Memory Safety<a class="anchor" aria-hidden="true" hidden href="#memory-safety">#</a></h4>
<p>There are mainly two notions of memory safety:</p>
<p><strong>Temporal safety</strong>:</p>
<ul>
<li>No use-after-free errors</li>
<li>Meaning, every pointer should point to a valid memory address.</li>
</ul>
<p><strong>Spatial safety</strong>:</p>
<ul>
<li>No out-of-bounds errors</li>
<li>Meaning, every pointer should point to a valid memory address within the allocated memory region.
<ul>
<li>It should not be able to read or write where you should not.</li>
</ul>
</li>
</ul>
<p>Most of the bugs are memory safety errors. Programs like rust solve this problem, but it is often infeasible to rewrite large codebases in memory safe languages like Rust.
Other fixes are:</p>
<ul>
<li><strong>Software fault isolation</strong>
<ul>
<li>Example: out of bound checks  on each memory access, WebAssembly does it.</li>
</ul>
</li>
<li><strong>Hardware support for memory safety</strong>:
<ul>
<li>Memory capabilities or protection keys see <a href="/notes/sicurezza-os/">Sicurezza OS</a>.</li>
<li>Idea -> <em>fat pointer</em>, that also has information about read-write range permissions.</li>
<li>One example is CHERI (university of Cambridge 2010 with ARM), now we have the first chips that have this feature.</li>
</ul>
</li>
</ul>
<h3 id="trust-relations">Trust Relations<a class="anchor" aria-hidden="true" hidden href="#trust-relations">#</a></h3>
<p>Trust relationships define our threat model, so it is important to know some of the most common treat models in the cloud and in enterprise systems.</p>
<h4 id="trust-in-the-cloud">Trust in the cloud<a class="anchor" aria-hidden="true" hidden href="#trust-in-the-cloud">#</a></h4>
<p>Trust relationships</p>
<ul>
<li>
<p>Everyone trusts the hardware</p>
</li>
<li>
<p>Users do not trust each other</p>
</li>
<li>
<p>Users trust the cloud provider (and its software)</p>
</li>
<li>
<p>Cloud provider does not trust users
Who has access to what?</p>
</li>
<li>
<p>Users do not have physical access to hardware</p>
</li>
<li>
<p>Cloud provider has physical access</p>
</li>
<li>
<p>VM isolation is the de-facto standard</p>
<ul>
<li>Even when the user doesn’t see/manage the VM (functions, containers)</li>
</ul>
</li>
<li>
<p>Some notable exceptions:</p>
<ul>
<li>Bare metal servers</li>
<li>gVisor (Google “application kernel” for Linux)</li>
<li>Drawbridge (Microsoft process-level isolation for Windows applications with a library OS)</li>
<li>Cloudflare workers (language sandbox for serverless functions, based on V8 JIT)</li>
</ul>
</li>
<li>
<p>Why VMs?</p>
<ul>
<li>Perception of greater security</li>
<li>Lowest common denominator when mitigating security vulnerabilities</li>
<li>Overhead has come down (e.g., AWS firecracker (<a href="/notes/cloud-computing-services/">Cloud Computing Services</a>))</li>
</ul>
</li>
</ul>
<h4 id="trust-in-enterprise-systems">Trust in Enterprise systems<a class="anchor" aria-hidden="true" hidden href="#trust-in-enterprise-systems">#</a></h4>
<p>Trust relationships</p>
<ul>
<li>Everyone trusts the hardware</li>
<li>Users do not trust each other</li>
<li>Users trust the enterprise (administrators etc.)</li>
<li>Enterprise does not trust users
Who has access to what?</li>
<li>Users have physical access to hardware</li>
<li>Enterprise has physical access</li>
</ul>
<h3 id="side-channel-and-covert-channel">Side Channel and Covert Channel<a class="anchor" aria-hidden="true" hidden href="#side-channel-and-covert-channel">#</a></h3>
<h4 id="definitions">Definitions<a class="anchor" aria-hidden="true" hidden href="#definitions">#</a></h4>
<ul>
<li>A side channel is a source of unintended communication between <strong>isolated</strong> components
<ul>
<li>Usually done with instruction that leak timing information, or hardware performance features, like spectre.</li>
<li>Here the leak is <strong>unintentional</strong> while with covert channels, it is intentional (see <a href="#dennettIntentionalStanceTheory1988">(Dennett 1988)</a> for intentionality).</li>
</ul>
</li>
<li>Covert channel: hidden channel used to get information out
<ul>
<li>Two parties want to communicate without others knowing the communication is happening</li>
</ul>
</li>
</ul>
<h4 id="other-types-of-attacks-with-side-channels">Other Types of attacks with side channels<a class="anchor" aria-hidden="true" hidden href="#other-types-of-attacks-with-side-channels">#</a></h4>
<p><img src="/images/notes/Notions of Security-20250527223226223.webp" style="width: 100%" class="center" alt="Notions of Security-20250527223226223"></p>
<h4 id="spectre-attack">Spectre Attack<a class="anchor" aria-hidden="true" hidden href="#spectre-attack">#</a></h4>
<p>This type of attack is enabled by speculation mechanisms (branch prediction) in the CPU hardware.
It is very hard to fix without degrading the performance, and changing hardware is quite costly.
They basically show that with VM colocation, they are able to read kernle memory of the other VM.</p>
<p><strong>Structure of the Attack</strong>
Set some microarchitectural state, for example:</p>
<ul>
<li>Clear 256 (these are the characters in ASCII code) cache blocks at a certain address</li>
<li>Prime a branch to taken state and read some violated memory. The idea is very simple.</li>
<li>Then you can try to extract the cache values, if there is a hit, then it is faster (slightly faster) and you know that that was the byte read.</li>
</ul>
<h4 id="fixes-to-spectre">Fixes to spectre<a class="anchor" aria-hidden="true" hidden href="#fixes-to-spectre">#</a></h4>
<ul>
<li>Never share at the same time the same CPU (so L1 or L2 cache) to different VMs using multithreading for example.</li>
<li>Sequential sharing is ok, you just do context switch.</li>
<li>Last level cache can be shared, the attack to this cache level is much more difficult.</li>
</ul>
