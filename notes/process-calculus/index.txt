27:T2622c,<h2 id="process-calculus-and-session-types-honda-yoshida-carbone">Process Calculus and Session Types (Honda, Yoshida, Carbone)<a class="anchor" aria-hidden="true" hidden href="#process-calculus-and-session-types-honda-yoshida-carbone">#</a></h2>
<h3 id="historical-lineage-and-motivation">Historical Lineage and Motivation<a class="anchor" aria-hidden="true" hidden href="#historical-lineage-and-motivation">#</a></h3>
<p>The process calculus literature on session types originates from the fundamental insight that <strong>communication itself can be typed</strong>. Just as data types codify the structure of data, session types codify the structure of communication and make it available to programming tools. The key intellectual trajectory runs:</p>
<ol>
<li><strong>Milner et al. (1992)</strong>: The π-calculus — a calculus of mobile processes with name-passing and channel mobility.</li>
<li><strong>Honda (1993)</strong>: "Types for Dyadic Interaction" at CONCUR'93 — the first session type system.</li>
<li><strong>Takeuchi, Honda &#x26; Kubo (1994)</strong>: An interaction-based language with typing at PARLE'94.</li>
<li><strong>Honda, Vasconcelos &#x26; Kubo (1998)</strong>: The canonical binary session type system at ESOP'98 (awarded ETAPS Test-of-Time Award, 2019).</li>
<li><strong>Honda, Yoshida &#x26; Carbone (2008/2016)</strong>: Multiparty Asynchronous Session Types (MPST) at POPL'08, journal version in JACM 2016.</li>
</ol>
<p>The driving application domain is <strong>structured communication-centred programming</strong>: web services, business protocols, parallel scientific computing, multi-core programming. The thing you have to remember is that session types are <em>not</em> just about preventing type mismatches on messages — they statically enforce the entire conversational protocol.</p>
<h4 id="the-π-calculus-as-foundation">The π-Calculus as Foundation<a class="anchor" aria-hidden="true" hidden href="#the-π-calculus-as-foundation">#</a></h4>
<blockquote>
<p>The π-calculus is a process algebra where channel names can be communicated over channels, enabling dynamic reconfiguration of communication topologies.</p>
</blockquote>

<h4 id="why-the-untyped-π-calculus-is-insufficient">Why the Untyped π-Calculus is Insufficient<a class="anchor" aria-hidden="true" hidden href="#why-the-untyped-π-calculus-is-insufficient">#</a></h4>
<p>The raw π-calculus is wildly nondeterministic and admits:</p>
<ul>
<li><strong>Communication mismatches</strong>: a process sends an integer where a string is expected.</li>
<li><strong>Arity mismatches</strong>: sender transmits 3 values, receiver expects 2.</li>
<li><strong>Protocol violations</strong>: a process sends when it should receive.</li>
<li><strong>Deadlocks</strong>: circular dependencies where everyone waits.</li>
<li><strong>Races</strong>: multiple processes compete for the same channel endpoint.</li>
</ul>
<p>Session types address all of these statically.</p>

<h3 id="multiparty-asynchronous-session-types-honda-yoshida-carbone-20082016">Multiparty Asynchronous Session Types (Honda-Yoshida-Carbone, 2008/2016)<a class="anchor" aria-hidden="true" hidden href="#multiparty-asynchronous-session-types-honda-yoshida-carbone-20082016">#</a></h3>
<p>This is the major technical contribution of the Honda-Yoshida-Carbone (HYC) line. The core insight: <strong>binary session types compose poorly for multi-party protocols</strong> — composing pairwise binary sessions can miss global invariants. The solution is to specify the protocol globally and then project to local types.</p>
<h4 id="design-methodology">Design Methodology<a class="anchor" aria-hidden="true" hidden href="#design-methodology">#</a></h4>
<blockquote>
<p>First, design and agree upon a <strong>global type</strong> <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathnormal">G</span></span></span></span> as an intended conversation scenario. A team of programmers then develops code, one for each participant, incrementally validating its conformance to the projections of <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathnormal">G</span></span></span></span>. When programs are executed, their interactions automatically follow the stipulated scenario.</p>
</blockquote>
<p>The workflow:</p>
<ol>
<li><strong>Specify</strong> a global type <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathnormal">G</span></span></span></span> (the choreography).</li>
<li><strong>Project</strong> <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6833em;"></span><span class="mord mathnormal">G</span></span></span></span> onto each participant <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.625em;vertical-align:-0.1944em;"></span><span class="mord mathnormal">p</span></span></span></span> to obtain local types <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.8867em;vertical-align:-0.1944em;"></span><span class="mord mathnormal">G</span><span class="mspace" style="margin-right:0.2778em;"></span><span class="mrel amsrm">↾</span><span class="mspace" style="margin-right:0.2778em;"></span></span><span class="base"><span class="strut" style="height:0.625em;vertical-align:-0.1944em;"></span><span class="mord mathnormal">p</span></span></span></span>.</li>
<li><strong>Type-check</strong> each participant's code against its local type.</li>
<li><strong>Compose</strong> — the global type guarantees that the composition is safe.</li>
</ol>
<p>This is a <strong>top-down</strong> approach to concurrent system design, in contrast to bottom-up composition of binary sessions.</p>

<h4 id="fundamental-properties">Fundamental Properties<a class="anchor" aria-hidden="true" hidden href="#fundamental-properties">#</a></h4>
<p>The MPST system guarantees three properties for well-typed programs:</p>
<ol>
<li><strong>Communication Safety</strong>: no process sends/receives on a channel with a mismatched type or to/from a wrong participant.</li>
<li><strong>Session Fidelity</strong>: the actual sequence of communications conforms to the global type.</li>
<li><strong>Progress</strong> (partial, strengthened by Coppo et al. 2016): if a session is ready to communicate, it can always make progress (no deadlock within a single session).</li>
</ol>
<blockquote>
<p><strong>Warning</strong>: The original POPL'08 paper had limited progress results. The full <strong>global progress</strong> theorem — that well-typed systems with dynamically interleaved sessions are deadlock-free — was established by Coppo, Dezani-Ciancaglini, Yoshida &#x26; Padovani (2016) via a rational reconstruction of the MPST system.</p>
</blockquote>
<h4 id="limitations-of-projection">Limitations of Projection<a class="anchor" aria-hidden="true" hidden href="#limitations-of-projection">#</a></h4>
<p>Not all intuitively correct global protocols are projectable:</p>
<ul>
<li>The merging condition can fail when different branches require different behaviours from uninvolved participants.</li>
<li><strong>Insufficient "knowledge of choice"</strong>: a participant not involved in the choice may need to know which branch was taken (requiring explicit notification messages).</li>
<li>Castagna, Dezani-Ciancaglini &#x26; Padovani (2011) studied this systematically and showed that the projectability problem is non-trivial, identifying classes of <strong>well-formed global types</strong> whose projections need no covert channels.</li>
</ul>
<p>See choreography projection for ongoing work on relaxing projectability.</p>

<h4 id="wadler-2012-classical-linear-logic-cp-and-gv">Wadler (2012): Classical Linear Logic (CP and GV)<a class="anchor" aria-hidden="true" hidden href="#wadler-2012-classical-linear-logic-cp-and-gv">#</a></h4>
<p>Wadler's "Propositions as Sessions" (ICFP'12) presents:</p>
<ul>
<li><strong>CP</strong>: a session-typed process calculus corresponding to <strong>classical</strong> linear logic (one-sided sequents, Girard 1987).</li>
<li><strong>GV</strong>: a linear functional language with session types.</li>
<li>A translation from GV into CP, formalizing the connection between standard session types and linear logic.</li>
</ul>
<p>The classical setting gives a cleaner duality (<span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.8491em;"></span><span class="mord"><span class="mord mathnormal">A</span><span class="msupsub"><span class="vlist-t"><span class="vlist-r"><span class="vlist" style="height:0.8491em;"><span style="top:-3.063em;margin-right:0.05em;"><span class="pstrut" style="height:2.7em;"></span><span class="sizing reset-size6 size3 mtight"><span class="mord mtight">⊥</span></span></span></span></span></span></span></span></span></span></span>) matching session type duality <span class="katex"><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.8201em;"></span><span class="mord accent"><span class="vlist-t"><span class="vlist-r"><span class="vlist" style="height:0.8201em;"><span style="top:-3em;"><span class="pstrut" style="height:3em;"></span><span class="mord mathnormal" style="margin-right:0.0576em;">S</span></span><span style="top:-3.2523em;"><span class="pstrut" style="height:3em;"></span><span class="accent-body" style="left:-0.1667em;"><span class="mord">ˉ</span></span></span></span></span></span></span></span></span></span>. The key payoff: <strong>deadlock freedom follows directly from the correspondence to linear logic</strong>, since cut elimination in CLL is confluent and terminating.</p>
<p><strong>Limitation</strong>: both Caires-Pfenning and Wadler's systems forbid <strong>cyclic process topologies</strong> (tree-shaped channel graphs only), which excludes many practical concurrent patterns. This was partially addressed by Dardha &#x26; Gay (2018) using <strong>priority annotations</strong> (à la Kobayashi).</p>
<h3 id="advanced-topics-and-extensions">Advanced Topics and Extensions<a class="anchor" aria-hidden="true" hidden href="#advanced-topics-and-extensions">#</a></h3>
<h4 id="structured-interactional-exceptions-carbone-honda-yoshida-2008">Structured Interactional Exceptions (Carbone, Honda, Yoshida 2008)<a class="anchor" aria-hidden="true" hidden href="#structured-interactional-exceptions-carbone-honda-yoshida-2008">#</a></h4>
<p>Exception handling in session types. The challenge: if one participant throws an exception mid-session, all other participants must be notified and transition consistently. HYC proposed <strong>structured exceptions</strong> that extend global types with try-catch blocks, ensuring that exception propagation respects the multiparty protocol structure.</p>
<h4 id="asynchronous-subtyping-for-mpst">Asynchronous Subtyping for MPST<a class="anchor" aria-hidden="true" hidden href="#asynchronous-subtyping-for-mpst">#</a></h4>
<p>Subtyping in the multiparty asynchronous setting is more complex than the binary synchronous case:</p>
<ul>
<li>The standard subtyping (Gay &#x26; Hole 2005) must account for <strong>message reordering</strong> in FIFO buffers.</li>
<li>Mostrous &#x26; Yoshida (2009) studied <strong>global principal typing</strong> in partially commutative asynchronous sessions.</li>
<li>The decidability of asynchronous session subtyping remains a subtle open question in general.</li>
</ul>
<h4 id="parameterised-and-indexed-session-types">Parameterised and Indexed Session Types<a class="anchor" aria-hidden="true" hidden href="#parameterised-and-indexed-session-types">#</a></h4>
<p>Standard MPST require a <strong>fixed</strong> number of participants determined at session start. Extensions:</p>
<ul>
<li><strong>Parameterised MPST</strong> (Charalambides, Dinges, Agha 2012): global types parameterised over the number of participants, enabling ring or pipeline topologies of arbitrary size.</li>
<li><strong>Indexed session types</strong>: types indexed by data values (dependent session types, Toninho-Caires-Pfenning 2011).</li>
</ul>
<h4 id="from-theory-to-practice-scribble-and-language-implementations">From Theory to Practice: Scribble and Language Implementations<a class="anchor" aria-hidden="true" hidden href="#from-theory-to-practice-scribble-and-language-implementations">#</a></h4>
<p>The MPST theory has been implemented in several settings:</p>
<ul>
<li><strong>Scribble</strong> (scribble.org): a protocol description language based on global types, used in industry (Red Hat, Ocean Observatories Initiative).</li>
<li><strong>Session Java</strong> (Hu, Yoshida, Honda 2008): session-based distributed programming in Java via code generation from Scribble protocols.</li>
<li><strong>Links</strong> (Gay, Vasconcelos et al.): session types in a web programming language.</li>
<li><strong>Haskell</strong>, <strong>F#</strong>, <strong>Rust</strong>, <strong>Go</strong>: various embeddings and libraries.</li>
</ul>

<h3 id="key-design-principles-summary">Key Design Principles (Summary)<a class="anchor" aria-hidden="true" hidden href="#key-design-principles-summary">#</a></h3>
<ul>
<li><strong>Linearity of session channels</strong>: each session endpoint is used exactly once per communication step. This prevents races and ensures protocol adherence. See linear types.</li>
<li><strong>Duality / Coherence</strong>: binary types use duality; multiparty types use projection from a global type ensuring all local types are mutually consistent.</li>
<li><strong>Separation of shared and session channels</strong>: shared channels (for session initiation, replicable) vs. session channels (linear, protocol-governed).</li>
<li><strong>Top-down design via global types</strong>: specify the whole protocol first, then project — analogous to choreography vs. orchestration in web services.</li>
<li><strong>Asynchronous FIFO semantics</strong>: models realistic networking; requires careful handling of message ordering and causal dependencies.</li>
</ul>
<h3 id="open-problems-and-active-research-directions">Open Problems and Active Research Directions<a class="anchor" aria-hidden="true" hidden href="#open-problems-and-active-research-directions">#</a></h3>
<ul>
<li><strong>Decidability of asynchronous multiparty subtyping</strong> in full generality.</li>
<li><strong>Dynamic session membership</strong>: participants joining/leaving mid-session.</li>
<li><strong>Integration with effects and state</strong>: combining session types with mutable state, exceptions, and IO.</li>
<li><strong>Automata-theoretic foundations</strong>: replacing global types with more expressive Protocol State Machines (Stutz &#x26; D'Osualdo, 2025) to overcome projectability limitations.</li>
<li><strong>Gradual session types</strong> (Igarashi, Thiemann, Vasconcelos, Wadler 2019): mixing static and dynamic checking for session types, enabling incremental adoption.</li>
<li><strong>Multiparty session types for AI agents</strong>: applying MPST to structure and verify communication protocols in multi-agent systems. See multi-agent AI safety, Ricardian contracts.</li>
