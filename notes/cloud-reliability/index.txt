2b:Tb269,<blockquote>
<p>Reliability is the ability of a system to remain <strong>operational</strong> over time, i.e., to offer the service it was designed for.</p>
</blockquote>
<p>Cloud Hardware and software fails. In this note, we will try to find methods to analyze and predict when components fail, and how we can prevent this problem.</p>
<h3 id="defining-the-vocabulary">Defining the vocabulary<a class="anchor" aria-hidden="true" hidden href="#defining-the-vocabulary">#</a></h3>
<h4 id="reliability-and-factors-of-influence">Reliability and Factors of Influence<a class="anchor" aria-hidden="true" hidden href="#reliability-and-factors-of-influence">#</a></h4>
<p>Reliability is the probability that a system will perform its intended function without failure over a specified period of time.
There are many factors that influence this value:</p>
<ul>
<li>Errors: system is operational, but it produces wrong results, inducing unexpected results..</li>
<li>Faults: some part of the system is not working, and as a result some of its functionality might be compromised.</li>
<li>Failures: system is not working, it needs to be repaired.</li>
<li>Performance issues: system is operational, but it is not performing as expected, e.g. too slow.</li>
</ul>
<h4 id="detecting-and-predicting-failures">Detecting and predicting failures<a class="anchor" aria-hidden="true" hidden href="#detecting-and-predicting-failures">#</a></h4>
<p>In usual Cloud system we <strong>monitor</strong> the system to gather information about its health:</p>
<ul>
<li><strong>Data collection</strong>: activity, errors, configuration about each node.</li>
<li><strong>Storage:</strong> We store the data plus some metadata (usually as time series).</li>
<li><strong>Prediction</strong> use some Machine Learning tools to predict these kinds of failures, so that we can proactively solve them (instead of just reactively).</li>
</ul>
<p>Then in order to hide the failures, every part of the stack should provide high availability and redundancy: not just software but also:</p>
<ul>
<li>Power and cooling</li>
<li>Network infrastructure</li>
<li>Scheduling, see <a href="/notes/cluster-resource-management/">Cluster Resource Management</a>.</li>
<li>Storage.</li>
</ul>

<h4 id="mttf-mean-time-to-failure">MTTF: Mean Time To Failure<a class="anchor" aria-hidden="true" hidden href="#mttf-mean-time-to-failure">#</a></h4>
<p>The failure rate is the number of faults per unit time.

<h4 id="best-effort">Best Effort<a class="anchor" aria-hidden="true" hidden href="#best-effort">#</a></h4>
<p>In some applications, we do not want the service to be down, but it would be ok to be inconsistent (remember  <a href="/notes/cloud-storage/#cap-theorem">Cloud Storage#CAP Theorem</a>).
For example, the google search could be a best effort application, here we do not offer service guarantees, but do the best to do it.</p>
<h3 id="providing-fault-tolerance">Providing Fault Tolerance<a class="anchor" aria-hidden="true" hidden href="#providing-fault-tolerance">#</a></h3>
<p>We need some form of replication for better reliability and error correction. Sometimes <strong>sharding</strong> is done (remember <a href="/notes/distributed-file-systems/">Distributed file systems</a>).</p>
<h4 id="data-striping">Data Striping<a class="anchor" aria-hidden="true" hidden href="#data-striping">#</a></h4>
<p>The idea is to distribute data among several disks so that it can be <strong>accessed in parallel</strong>, this is usually implemented at RAID level, see <a href="/notes/devices-os/#raid">Devices OS#RAID</a>, level 0.
This is a easier way to decrease latency and increase throughput.</p>
<h4 id="fine-and-coarse-grained-striping">Fine and Coarse Grained Striping<a class="anchor" aria-hidden="true" hidden href="#fine-and-coarse-grained-striping">#</a></h4>
<p>Fine-grain and coarse-grain data striping are techniques used in parallel computing and storage systems to distribute data across multiple devices to improve performance and fault tolerance.</p>
<p>Fine-grained:</p>
<ul>
<li><strong>Definition</strong>: Data is divided into very small chunks and distributed across multiple disks or processing units.</li>
<li><strong>Advantages</strong>:
<ul>
<li>Improves load balancing, as all disks contribute to every I/O request.</li>
<li>Provides high parallelism, making it well-suited for applications with many small, frequent data accesses.</li>
</ul>
</li>
<li><strong>Disadvantages</strong>:
<ul>
<li>High overhead due to increased coordination among disks.</li>
<li>May not be optimal for large sequential reads and writes.</li>
</ul>
</li>
<li><strong>Example</strong>: RAID 3, where data is split at the byte or bit level across disks, requiring all disks to participate in every operation.</li>
</ul>
<p>Coarse-grained:</p>
<ul>
<li><strong>Definition</strong>: Data is divided into larger blocks before being distributed across multiple disks or processing units.</li>
<li><strong>Advantages</strong>:
<ul>
<li>Reduces overhead by allowing each disk to handle independent requests.</li>
<li>Better suited for workloads with large sequential data accesses.</li>
</ul>
</li>
<li><strong>Disadvantages</strong>:
<ul>
<li>Less effective load balancing for small requests since fewer disks participate per operation.</li>
<li>May result in uneven disk utilization.</li>
</ul>
</li>
<li><strong>Example</strong>: RAID 5, where data is striped at the block level, allowing parallel access without requiring all disks to be involved in every operation.</li>
</ul>

<h4 id="sharding">Sharding<a class="anchor" aria-hidden="true" hidden href="#sharding">#</a></h4>
<p>We can choose to have big or small shards.</p>
<ul>
<li><strong>Big shards</strong>: they are easier to manage, but if they fail, we lose a lot of data, or lots of data could be on the same disk creating a <strong>bottleneck</strong>, also called <strong>hotspot</strong> (if you have a lot of clients that attempt to read from this shard).</li>
<li><strong>Small shards</strong>: they are harder to manage, since they are more, but they are more fault tolerant and have a better load balancing.</li>
</ul>

<h3 id="placing-the-blocks">Placing the blocks<a class="anchor" aria-hidden="true" hidden href="#placing-the-blocks">#</a></h3>
<p>This section mainly explores ideas presented in <a href="https://www.usenix.org/conference/atc13/technical-sessions/presentation/cidon">(Cidon et al. 2013)</a>.</p>
<h4 id="random-placing">Random Placing<a class="anchor" aria-hidden="true" hidden href="#random-placing">#</a></h4>
<p>Random block placement is a good idea, if we assume failures of blocks is <strong>uncorrelated</strong> with each other.
But in reality it seems that failures are correlated (see paper <a href="https://www.usenix.org/conference/atc13/technical-sessions/presentation/cidon">(Cidon et al. 2013)</a>), for example you could have some rack failures or network failures.</p>
<p>Frameworks like <a href="/notes/distributed-file-systems/">Distributed file systems</a> HDFS do it one in the node that receives, one in the same rake, and other one in another node in the same cluster, which means they choose a <strong>deterministic distribution</strong>.</p>
<p>The paper shows that the probability of losing data becomes very high if you use random placing and you have many nodes (around 1000 nodes, you have almost 100%).</p>
<h4 id="copyset-replication">Copyset Replication<a class="anchor" aria-hidden="true" hidden href="#copyset-replication">#</a></h4>
<p>A copyset (see paper <a href="https://www.usenix.org/conference/atc13/technical-sessions/presentation/cidon">(Cidon et al. 2013)</a>) is a set of nodes that contain all the blocks of the data. Then you do random replication. But this is a single point of failure, so you do random replication. But losing three servers makes you lose some data.</p>
<p>Copysets says: if I place a datablock to one node, then it <strong>must</strong> be placed in the same copyset as the other copies of the same block.</p>
<ul>
<li>Place first block randomly</li>
<li>Then place following the copysets.
This leads to better reliability against losing data when we have correlated failures.
The problem here is that</li>
<li>when we lose data, we lose more data (all data),</li>
<li>another problem is that this method is slower in recovering data (since we need to read from the same servers).</li>
</ul>
<p>They made the argument that:</p>
<ul>
<li>Better to lose 1TB in 625 years than</li>
<li>Losing 1GB every year, using MTTF analysis as above.</li>
</ul>
<h3 id="other-reliability-factors">Other Reliability Factors<a class="anchor" aria-hidden="true" hidden href="#other-reliability-factors">#</a></h3>
<h4 id="load-balancing">Load Balancing<a class="anchor" aria-hidden="true" hidden href="#load-balancing">#</a></h4>
<p>Look at <a href="/notes/content-delivery-networks/">Content Delivery Networks</a> for more.
Load balancing goes well with sharding
Ability to balance load depends on the nature of jobs and data distribution</p>
<ul>
<li>Micro-service &#x26; micro-shard tendencies go together</li>
<li>the smaller the components, the more we can balance and distribute the load</li>
</ul>
<p>REST makes load balancing easier:
• Requests are independent, neither the client nor the server keep state (to some extent)
• Each request can be processed by a different machine:
• Higher throughput by adding more machines
• Higher resilience by adding more machines</p>
<h4 id="redundant-execution">Redundant Execution<a class="anchor" aria-hidden="true" hidden href="#redundant-execution">#</a></h4>
<p>This is a nice way to trim down tail latency.
Basically you send the same request twice, so that if one machine finishes first, then you have the answer faster. It reduces the problem of stragglers, explored in <a href="https://dl.acm.org/doi/10.1145/2408776.2408794">(Dean &#x26; Barroso 2013)</a>.</p>
<h4 id="monitoring-tools">Monitoring Tools<a class="anchor" aria-hidden="true" hidden href="#monitoring-tools">#</a></h4>
<p>Many other techniques involved in maintaining availability:</p>
<ul>
<li><strong>Canaries</strong>: a large scale parallel job that has a bug can trigger a massive failure; run the job in a “canary” and only run at scale if it has no problems. Comes from canary in coal mines, you first send synthetic requests to check if it works. -> <em>gradual rollout</em> of big systems.</li>
<li><strong>Watchdog timers</strong>: regularly check the responsiveness of a machine to identify the ones that may cause trouble</li>
<li><strong>Integrity checks</strong>: regular integrity checks that complement standard techniques in hardware to make sure that data is not lost or corrupted; run by background jobs</li>
</ul>
<p>Need good logging and analysis tools:</p>
<ul>
<li>ML is often applied today to better understand problems and help predict them</li>
</ul>
